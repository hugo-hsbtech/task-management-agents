# Product Analytics — Universal Patterns

Provider-agnostic patterns for designing, implementing, and governing product analytics events. Use alongside a provider skill for SDK-specific implementation.

## When to Activate

- Designing a new tracking event or property
- Setting up user identification in a new flow
- Auditing existing events for naming or governance issues
- Planning group analytics for a B2B product
- Reviewing analytics implementation in code

## The Event Model

Every analytics event has four components:

| Component | Description | Example |
|---|---|---|
| **Name** | What happened | `subscription_upgraded` |
| **Properties** | Context about what happened | `{ plan_from: 'free', plan_to: 'pro' }` |
| **User identity** | Who triggered it | `distinct_id: 'usr_123'` |
| **Timestamp** | When it happened | Auto-generated by the SDK |

Properties carry the context. The event name describes the action only.

## Event Naming Convention

Use `snake_case` with `object_past_verb` pattern — flat, no category prefixes.

```
{object}_{past_tense_verb}
```

```
# Good: specific object + past tense verb
user_signed_up
project_created
subscription_upgraded
payment_failed
invite_sent
dashboard_viewed
file_uploaded
member_removed

# Bad: present tense
user_sign_up
project_create

# Bad: vague
button_clicked
action_performed

# Bad: dynamic value in the name
pricing_page_viewed      # Good
/pricing-viewed          # Bad — URL in name, breaks queries
checkout_step_2_viewed   # Bad — encode step as a property instead
```

### Approved Verb List

Standardize verbs across the team. Never use synonyms for the same action.

| Action | Use | Avoid |
|---|---|---|
| Navigation | `viewed` | `opened`, `visited`, `navigated_to` |
| Creation | `created` | `added`, `made`, `generated` |
| Deletion | `deleted` | `removed`, `destroyed` |
| Form submit | `submitted` | `sent`, `saved`, `completed` |
| Invitation | `invited` | `added`, `shared` |
| Upgrade | `upgraded` | `changed`, `switched` |
| Sign up | `signed_up` | `registered`, `created` |
| Sign in | `signed_in` | `logged_in`, `authenticated` |
| Sign out | `signed_out` | `logged_out`, `disconnected` |

## Property Naming Convention

```
{noun}_{descriptor}       → plan_type, item_price, page_name
is_{state}                → is_trial, is_subscribed, is_admin
has_{thing}               → has_payment_method, has_team_members
{noun}_count              → member_count, file_count, api_call_count
{noun}_id                 → user_id, project_id, organization_id
{noun}_date               → trial_end_date, created_date
{noun}_at                 → signed_up_at, last_seen_at  (ISO 8601 timestamp)
```

Use primitive types only — strings, numbers, booleans. Avoid nested objects in properties.

```
# Good: flat properties
posthog.capture('subscription_upgraded', {
    plan_from: 'free',
    plan_to: 'pro',
    billing_interval: 'annual',
    upgrade_source: 'pricing_page',
})

# Bad: nested object — breaks most analytics UIs
posthog.capture('subscription_upgraded', {
    plan: { from: 'free', to: 'pro' },
})
```

## User Identity Lifecycle

```
anonymous → identify → set properties → [alias if needed] → reset on logout
```

### Anonymous Phase

The SDK auto-generates a `distinct_id` (UUID) for unidentified users. Never override this manually.

### Identification

Call `identify()` once — at sign-up or sign-in — with a **stable, permanent database ID**.

```
# Good: permanent database ID
identify('usr_abc123', { email: 'carlos@ezra.com' })

# Bad: email as ID — changes if user updates email
identify('carlos@ezra.com', {})

# Bad: username — can change
identify('carlosmelgoza', {})
```

### User Properties

Set properties that describe the user, not the event. These persist on the user profile.

| Property type | Use for | Example |
|---|---|---|
| `set` | Mutable attributes | `plan_type`, `role`, `company_name` |
| `set_once` | Immutable first-touch | `signed_up_at`, `acquisition_source`, `first_plan` |

Always set `set_once` properties at sign-up. They capture the state at acquisition and must not be overwritten.

### Reset on Logout

Always call `reset()` on logout. This generates a new anonymous ID for the next session and prevents cross-user data leakage on shared devices.

## Super Properties (Persistent Event Context)

Super properties attach automatically to every event without repeating them in each `capture()` call. Use for values that are always relevant:

- `app_version`
- `environment` (`production`, `staging`)
- `platform` (`web`, `mobile`, `api`)
- `plan_tier` (current user plan)

```
# Good: set once at init, fires on every event
analytics.register({
    app_version: '2.4.1',
    environment: 'production',
    platform: 'web',
})

# Bad: repeating context on every event call
analytics.capture('project_created', {
    app_version: '2.4.1',  # should be a super property
    environment: 'production',
    project_name: 'My Project',
})
```

## Group Analytics (B2B)

Groups let you attribute events to an organization or team, not just a user. Essential for B2B products where the buyer is a company.

```
# Assign user to a group
analytics.group('organization', 'org_456', {
    name: 'Acme Corp',
    plan: 'enterprise',
    member_count: 42,
})

# Group properties to set
organization_id        # stable DB ID — the group key
name                   # display name
plan                   # billing plan
member_count
created_at
industry
```

Events captured after `group()` are attributed to both the user and the organization.

Group key must be a stable database ID — never a display name that can change.

## Governance Rules

1. **No dynamic values in event names.** Use a property instead.
2. **One verb per action.** Pick from the approved verb list and never add synonyms.
3. **Every event needs an owner.** Someone on the team is responsible for each event's accuracy.
4. **Properties are additive.** Never remove a property from a live event — add new ones, deprecate old ones.
5. **Never track PII in event properties.** No full names, passwords, card numbers, or raw email addresses in properties.
6. **Test both client and server events.** Verify properties arrive correctly in the analytics dashboard before shipping.
7. **Keep event count manageable.** Aim for fewer, richer events rather than many narrow ones.

## Anti-Patterns

```
# Bad: dynamic value encoded in event name — creates thousands of unique events
analytics.capture(`${page_name}_viewed`)

# Good: one event, context in a property
analytics.capture('page_viewed', { page_name })
```

```
# Bad: tracking same action with different names in different places
analytics.capture('add_to_cart')      # checkout team
analytics.capture('item_added_cart')  # mobile team
analytics.capture('CartItemAdded')    # Amplitude convention mixed in

# Good: one agreed name used everywhere
analytics.capture('cart_item_added')
```

```
# Bad: identifying with mutable identifier
analytics.identify(user.email)

# Good: stable database ID
analytics.identify(user.id)
```

```
# Bad: not resetting on logout — next anonymous user shares identity
// user signs out
router.push('/login')

# Good: reset first, then redirect
analytics.reset()
router.push('/login')
```

**Remember**: Events are append-only in production. You cannot retroactively rename an event or remove a property without breaking historical charts. Get the naming right before shipping — it is far cheaper than a data migration.

## Platform Method Reference

| Concept | PostHog | Amplitude | Mixpanel |
|---|---|---|---|
| Track event | `posthog.capture()` | `amplitude.track()` | `mixpanel.track()` |
| Identify user | `posthog.identify()` | `amplitude.setUserId()` | `mixpanel.identify()` |
| User properties | `$set` / `$set_once` | `Identify.set()` | `people.set()` |
| Super properties | `posthog.register()` | No native equivalent | `mixpanel.register()` |
| Group analytics | `posthog.group()` | `amplitude.setGroup()` | `mixpanel.set_group()` |
| Reset on logout | `posthog.reset()` | `amplitude.reset()` | `mixpanel.reset()` |

Note: PostHog's primary method is `capture()`, not `track()` — the only outlier among the three.
