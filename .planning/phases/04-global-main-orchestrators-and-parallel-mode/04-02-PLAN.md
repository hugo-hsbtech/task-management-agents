<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Linear MCP `read` response → GlobalOrchestrator | Untrusted data from Linear MCP layer flows into filter/sort logic; mitigated by Pydantic `extra="forbid"` on output and defensive `.get()` access on incoming dict fields (no KeyError on missing keys) |
| Test mocks → CI runner | `patch.object(go, "_fetch_all_items", ...)` injects synthetic dicts during unit tests; mitigated by mocks staying in tests/ and never reaching production code paths |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-4-01 | Spoofing | GlobalOrchestrator output | accept | Global Orchestrator does not perform claiming — it only reads. Double-claim mitigation lives in Plan 03 Main Orchestrator. The output `ready_tasks` list is advisory; downstream code (Main Orchestrator) is responsible for the optimistic-lock claim before dispatch |
| T-4-02 | Tampering | _filter_ready_items input | mitigate | All field access uses `.get()` with safe defaults (priority=999, dependencies=[], qa_status=None default). Malicious or malformed Linear responses cannot crash the filter; they at most produce an empty `ready_tasks` list, which Main Orchestrator handles gracefully |
| T-4-03 | DoS | _fetch_all_items result size | accept | Linear MCP `list_issues` returns the full project scope. For HSBTech v1, project size is bounded (single project, single repo); accepted. Pagination/streaming deferred to v2 if needed |
| T-4-04 | Information Disclosure | logger.info call sites | mitigate | The class uses `logger = logging.getLogger(__name__)`. No log line in this file logs API keys or full Linear payloads — only counts and IDs (e.g., "Returning N ready tasks"). Audit during code review |
</threat_model>
