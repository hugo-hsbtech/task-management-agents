## 1. System Classification

**System Type:** Code Automation / Developer Tooling

**Description:**
Phase 1 builds the foundational layer of an internal multi-agent delivery pipeline. It establishes the Linear Agent — a pydantic-validated CLI tool that wraps the Linear MCP server (`mcp__linear__*` tools) to create, update, comment on, and link GitHub PRs to Linear work items. The operator (an HSBTech engineer running the pipeline) interacts via a typer CLI; every operation is schema-validated on input and output using pydantic models that mirror AGENT-CONTRACTS.md. "Good" looks like: a CLI invocation creates the correct hierarchy in Linear, every field validates, failures retry with exponential backoff, and the `updatedAt` optimistic-lock check fires correctly on writes.

**Critical Failure Modes:**
1. Linear write operations execute without pydantic validation, letting malformed payloads reach the MCP server and corrupt Linear state
2. Retry logic is absent or incorrect — a transient Linear API error causes a silent failure rather than a retried, logged operation
3. MCP tool names diverge from the confirmed `mcp__linear__*` prefix, causing tool-not-found errors at runtime that are silent during development
4. Pydantic models in `src/hsb/contracts/` diverge from AGENT-CONTRACTS.md schemas, causing downstream agents (Phase 2+) to receive structurally incorrect data
5. Knowledge Store directory is missing or incorrectly structured, breaking the Intelligence Agent's glob/grep retrieval in Phase 5

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer tooling / AI-assisted software delivery (internal operator tool)

**User Population:** HSBTech software engineers ("operators") who trigger the multi-agent delivery pipeline via CLI. These are technical practitioners who interpret structured output contracts, read audit logs, and diagnose failures at the MCP tool boundary. They are the sole users — there is no end-customer facing surface.

**Stakes Level:** High

**Output Consequence:** Every write the Linear Agent performs is a mutation to the system of record for the entire delivery pipeline. A malformed or incorrectly structured write (wrong parent linkage, missing required field, wrong `updatedAt` on optimistic lock) corrupts the Linear state that all downstream agents (Builder, QA, Git, Orchestrator) read on every cycle. Reverting Linear state is manual. A missed retry on a transient failure results in a silent gap in the work item hierarchy that is not discovered until a downstream agent attempts to read a missing entity.

### What Domain Experts Evaluate Against

---

**Dimension:** Contract fidelity — write output matches declared schema

Good (operator would accept): Every field in the LinearOutput contract is present, correctly typed, and matches AGENT-CONTRACTS.md exactly. The `id` field matches `^LIN-\d+$`. The `url` field starts with `https://linear.app/`. Pydantic `extra="forbid"` raises on any unknown field.

Bad (operator would flag): The agent returns a result where `linear_entities` is an empty list on a successful create, or the `id` field is a UUID instead of `LIN-\d+`, or an undeclared field appears in the envelope — all passing pydantic validation only because the model was not in strict mode.

Stakes: Critical

Source: Practitioner standard for contract-first API design; Pydantic v2 strict mode documentation; internal AGENT-CONTRACTS.md mandate (FOUND-03)

---

**Dimension:** Write atomicity — Linear state matches intended mutation after the call completes

Good (operator would accept): After a `create_issue` or `update_issue` call, the agent re-reads the entity via `mcp__linear__get_issue` and confirms the returned `updatedAt` is newer than the pre-write value. The re-read result is logged in the audit trail alongside the write payload.

Bad (operator would flag): The agent reports `result: "success"` without performing the post-write verification read. The operator has no way to confirm the write landed. If a concurrent modification occurred, the stale-write goes undetected.

Stakes: Critical

Source: Optimistic locking pattern (LINR-05); established reliability engineering practice for distributed write interfaces; PITFALLS.md Pitfall 4 (stale state)

---

**Dimension:** Retry correctness — transient failures are retried, permanent failures are surfaced

Good (operator would accept): A transient Linear API error (5xx, timeout) triggers the `PostToolUseFailure` hook with exponential backoff (1s, 2s, 4s). Each attempt is logged with attempt number and delay. After 3 failures the agent returns `result: "failed"` with `error_type: "tool_failure"` and the operator is informed immediately via CLI output.

Bad (operator would flag): A transient error causes the agent to silently return `result: "success"` with empty `linear_entities`, or to retry in a tight loop without delay (retry storm), or to exceed the 3-attempt limit and continue retrying indefinitely without surfacing to the operator.

Stakes: High

Source: Exponential backoff reliability engineering standard; LINR-05 acceptance criteria; MCP tool failure mode research (MCPEval, AWS developer tools failure mode taxonomy)

---

**Dimension:** MCP tool prefix correctness — tool names resolve at runtime

Good (operator would accept): At session init, the `SystemMessage(subtype="init")` `mcp_servers` field is read and the confirmed prefix is `mcp__linear__`. All tool calls in the session use this prefix. The SKILL.md and system prompt reference the same prefix. The operator sees `[TOOL] mcp__linear__create_issue` in CLI output.

Bad (operator would flag): The SKILL.md or system prompt references `mcp__claude_ai_Linear__create_issue` (the Claude.ai browser prefix). The tool call silently fails to resolve. The agent either loops requesting permission or produces output that looks correct but contains no actual Linear write.

Stakes: High

Source: Claude Agent SDK MCP guide; confirmed production failure mode documented in PITFALLS.md and Common Pitfalls section 1 of this spec; MCPEval research on tool name resolution failures

---

**Dimension:** Schema drift detection — pydantic models remain synchronized with AGENT-CONTRACTS.md

Good (operator would accept): When a field is added or renamed in AGENT-CONTRACTS.md, the corresponding pydantic model in `src/hsb/contracts/` raises a `ValidationError` on the first run with the old payload. The error message names the specific field and its expected type. The developer updates the model before any downstream agent can receive stale contract data.

Bad (operator would flag): A field is silently coerced (e.g., `"123"` accepted for an `int` field, or a `str` UUID accepted for a field declared `pattern=r"^LIN-\d+$"`). No error is raised. The pydantic model and AGENT-CONTRACTS.md diverge silently. Downstream agents in Phase 2+ receive structurally incorrect data. The divergence is only discovered during a QA cycle failure.

Stakes: High

Source: Pydantic v2 strict mode documentation; practitioner pattern "forbid extra fields + validate a sample early to detect schema drift" (Ranjan Kumar, Validation Layer Design); internal constraint FOUND-03

### Known Failure Modes in This Domain

1. **Silent type coercion masking schema violations.** Pydantic's default (lax) mode accepts `"123"` for `int`, `None` for optional fields that should be required in a given operation, and string UUIDs for fields with regex constraints — without raising. The contract appears valid but the Linear write receives a semantically wrong payload. Mitigation: use `model_config = {"extra": "forbid"}` on all contract models and validate id/url fields with explicit regex patterns.

2. **MCP tool name mismatch between development and runtime environments.** In Python `query()` sessions the tool prefix is determined by the key in `mcp_servers` (e.g., `"linear"` → `mcp__linear__`). In Claude.ai browser sessions the same Linear MCP appears as `mcp__claude_ai_Linear__`. If SKILL.md or the system prompt is written against the browser prefix, every tool call in the Python SDK context fails silently — the agent either loops or hallucinates a success response. This failure is invisible during prompt authoring and only manifests at runtime.

3. **Retry storm from unbounded or zero-delay backoff.** Without a `PostToolUseFailure` hook that enforces both a maximum retry count and a minimum delay, a transient Linear API error can produce a tight retry loop that (a) exhausts the `max_turns` budget, (b) floods the Linear API with redundant requests, or (c) causes the agent to declare success after a semantically failed tool call because the context window accumulated enough error messages that the model lost track of the failure state.

4. **Stale-read false positive on optimistic lock.** The `updatedAt` field must be read immediately before a write and compared to the post-write re-read. If the pre-write read is cached or was performed several tool-call turns earlier (during which another process mutated the issue), the optimistic lock comparison produces a false positive: the agent believes the write was clean but a concurrent mutation was silently overwritten.

## 2. Framework Decision

**Selected Framework:** Claude Agent SDK

**Version:** 0.1.73+ (pinned in pyproject.toml per FOUND-02)

**Rationale:**
Claude Agent SDK is explicitly mandated by FOUND-02. The mandate is architecturally correct: Phase 1 is MCP-heavy (Linear MCP via `mcp-remote`, `mcp__linear__*` tool prefix), uses SKILL.md auto-discovery for the Linear skill, and requires the 18 lifecycle hooks for retry/backoff instrumentation. No adapter layer is needed between the pydantic contract interface and the Linear MCP server. The SDK's safety-first design and automatic context compaction are beneficial for an internal tool that runs unattended within a larger delivery pipeline.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Direct Anthropic Python SDK (`anthropic>=0.40`) | Viable fallback if Agent SDK API proves too volatile; lacks native MCP integration and SKILL.md auto-discovery — would require manual tool-use loop wiring |
| LangChain / LangGraph | Explicitly excluded by REQUIREMENTS.md Out of Scope: "Conflicts with SKILL.md format; adds abstraction that doesn't compose with Claude Agent SDK" |
| CrewAI | Explicitly excluded by REQUIREMENTS.md Out of Scope (same reason); Python-only but coarse error handling conflicts with required exponential backoff contract |

**Vendor Lock-In Accepted:** Yes — Anthropic/Claude only. Trade-off documented: the architecture is built around Claude Agent SDK primitives (SKILL.md, MCP, lifecycle hooks). Model-agnosticism is not a goal for v1; operator-facing tooling for HSBTech only.

### Entry Point Pattern

Minimal working Linear Agent that creates a Linear issue via MCP tool call:

```python
"""
src/hsb/agents/linear_agent.py

Minimal Linear Agent entry point for Phase 1.
Wraps mcp__linear__* tools behind validated contract I/O.
Requires: ANTHROPIC_API_KEY env var, Linear MCP accessible via mcp-remote.
"""
import asyncio
import os
from claude_agent_sdk import (
    query,
    ClaudeAgentOptions,
    SystemMessage,
    AssistantMessage,
    ResultMessage,
)
from dotenv import load_dotenv

load_dotenv()  # Load ANTHROPIC_API_KEY from .env


async def run_linear_agent(prompt: str) -> str | None:
    """Execute one Linear Agent turn. Returns the result string or None on failure."""
    options = ClaudeAgentOptions(
        # Linear MCP server via mcp-remote transport (OAuth 2.1 handled by mcp-remote)
        mcp_servers={
            "linear": {
                "command": "npx",
                "args": ["-y", "mcp-remote", "https://mcp.linear.app/mcp"],
                # For headless/CI: pass API key via env instead of browser OAuth
                # "env": {"LINEAR_API_KEY": os.environ["LINEAR_API_KEY"]},
            }
        },
        # Explicitly allow the MCP tools the agent needs — wildcard is safe here
        # because mcp-remote only exposes the Linear server.
        # In production, enumerate specific tools: ["mcp__linear__create_issue", ...]
        allowed_tools=["mcp__linear__*"],
        # acceptEdits auto-approves filesystem tools; does NOT auto-approve MCP tools.
        # MCP tools are approved via allowed_tools above.
        permission_mode="acceptEdits",
        # System prompt sets the Linear Agent's behavioral contract
        system_prompt=(
            "You are the Linear Agent for the HSBTech AI Engineering Workflow. "
            "You manage Linear work items via the mcp__linear__* tools. "
            "You MUST validate all inputs against the contract schema before calling tools. "
            "On tool failure, retry up to 3 times with exponential backoff (1s, 2s, 4s). "
            "Always confirm the write was successful by re-reading the updated entity."
        ),
    )

    result_text = None

    async for message in query(prompt=prompt, options=options):
        # Capture session ID from init for future session resumption
        if isinstance(message, SystemMessage) and message.subtype == "init":
            session_id = message.data.get("session_id")
            mcp_servers = message.data.get("mcp_servers", [])
            failed = [s for s in mcp_servers if s.get("status") != "connected"]
            if failed:
                raise RuntimeError(f"Linear MCP server failed to connect: {failed}")

        # Stream assistant reasoning to terminal for operator visibility
        elif isinstance(message, AssistantMessage):
            for block in message.content:
                if hasattr(block, "text"):
                    print(block.text)
                elif hasattr(block, "name"):
                    # MCP tool call: confirm prefix is mcp__linear__
                    print(f"[TOOL] {block.name}")

        # Capture final result
        elif isinstance(message, ResultMessage):
            if message.subtype == "success":
                result_text = message.result
            else:
                raise RuntimeError(f"Agent failed: {message.subtype}")

    return result_text


if __name__ == "__main__":
    # Example: create a Linear issue
    result = asyncio.run(
        run_linear_agent(
            "Create a Linear issue in the HSBTech project with title "
            "'[TASK] Phase 1 verification' and description "
            "'Verify Linear MCP connection is operational.'"
        )
    )
    print(result)
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `query()` | Async iterator that runs the full agent loop — tool execution, retries, context management are all automatic. Returns `SystemMessage`, `AssistantMessage`, `ResultMessage` objects. | Every agent invocation. `async for message in query(prompt=..., options=...)` is the only entry point. |
| `ClaudeAgentOptions` | Configuration object. Controls which tools are allowed, which MCP servers connect, permission mode, system prompt, hooks, and session resumption. | Passed to `query()` on every call. Build once per agent type, reuse across calls. |
| `mcp_servers` dict | Maps server name to transport config (`command`/`args` for stdio, `type`/`url` for HTTP/SSE). The server name becomes the `mcp__<name>__` prefix in tool names. | Wire the Linear MCP server. Key `"linear"` produces tool prefix `mcp__linear__`. |
| `allowed_tools` list | Whitelist of tool names Claude may call. MCP tools require explicit `mcp__<server>__<tool>` entries or wildcard `mcp__<server>__*`. Without this, Claude sees tools but cannot call them. | Required for any MCP tool access. `permission_mode="acceptEdits"` does NOT cover MCP tools. |
| `HookMatcher` + hooks dict | Register Python async callbacks on named lifecycle events (`PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `Stop`, `SubagentStart`, `SubagentStop`, `PreCompact`, `Notification`, and others). Matchers are regex against tool name (`^mcp__linear__` matches all Linear tools). | Implement retry/backoff instrumentation, audit logging, and block operations on validation failure. |
| `AgentDefinition` | Named subagent with its own `prompt`, `tools`, and `description`. The parent agent invokes it via the `Agent` tool. | Phase 2+: define specialized agents (Builder, QA, Git). Not needed in Phase 1. |
| Session resumption (`resume=session_id`) | Pass a captured `session_id` from `SystemMessage(subtype="init")` to `ClaudeAgentOptions(resume=...)` to continue a session across CLI invocations. State is stored as JSONL on the filesystem. | Persist context across `run-next-step` CLI calls. Capture `session_id` in `.claude/session_cache.json`. |

### Common Pitfalls

1. **`mcp__linear__` vs `mcp__claude_ai_Linear__` prefix confusion.** In SDK-managed sessions (Python `query()`), MCP tools use the key you define in `mcp_servers` — e.g., key `"linear"` produces `mcp__linear__create_issue`. In Claude.ai browser sessions, the same Linear MCP may appear as `mcp__claude_ai_Linear__create_issue`. These are incompatible. If your SKILL.md or prompt refers to the wrong prefix, the tool-not-found error is silent at development time and only fails at runtime. Always verify the prefix against the `SystemMessage(subtype="init")` `mcp_servers` field before building prompts.

2. **`permission_mode="acceptEdits"` does not grant MCP tool access.** This is the most common misconfiguration. `acceptEdits` auto-approves file edits and filesystem Bash commands only. MCP tools (`mcp__linear__*`) require explicit entries in `allowed_tools`. If `allowed_tools` is missing or uses the wrong prefix, Claude receives tool definitions but is blocked from calling them — the agent will either loop asking for permission or silently skip the tools.

3. **OAuth token lifecycle with `mcp-remote`.** The first `query()` call that includes the Linear MCP server will open a browser tab for OAuth 2.1 authorization if no cached token exists. This is interactive-only — it blocks a headless/CI run indefinitely. For CI or automated use: (a) pass `LINEAR_API_KEY` via the server's `env` field instead of relying on OAuth, or (b) pre-warm the OAuth token by running one interactive session and caching it. The token cache lives at `~/.mcp-remote/` by default; verify its presence before assuming headless auth works.

4. **Context window overflow on large Linear workspace queries.** `mcp__linear__list_issues` without filters can return hundreds of issues. Each issue is injected into Claude's context window. At scale, a single un-filtered list call can exhaust the context budget mid-session, triggering auto-compaction and silently dropping earlier contract instructions. Always pass filters: `teamId`, `projectId`, `stateId`, or `limit`. In Phase 1, the Linear Agent should never call `list_issues` without an explicit scope constraint.

5. **`asyncio.run()` called inside a running event loop.** The Agent SDK is async-first. If you wrap `asyncio.run(run_linear_agent(...))` inside a Typer callback that is itself launched from an async context (e.g., Jupyter, FastAPI, or another async harness), Python raises `RuntimeError: This event loop is already running`. Use `anyio.from_thread.run_sync()` or restructure CLI entry points to be top-level `asyncio.run()` calls only. Typer is synchronous by default — safe to use with `asyncio.run()` at the CLI boundary, but do not nest.

**Core Pattern — Retry/Backoff Hook for Linear MCP Calls:**

```python
"""
src/hsb/agents/hooks.py

PostToolUseFailure hook: implements exponential backoff for mcp__linear__* tool failures.
Wires into ClaudeAgentOptions(hooks={...}).
"""
import asyncio
from claude_agent_sdk import HookMatcher


# Track retry counts per tool_use_id across hook invocations
_retry_counts: dict[str, int] = {}
MAX_RETRIES = 3
BASE_DELAY_SECONDS = 1.0


async def linear_retry_hook(input_data: dict, tool_use_id: str | None, context) -> dict:
    """
    PostToolUseFailure hook for mcp__linear__* tools.
    Implements exponential backoff and injects retry instruction into conversation.
    """
    tool_name = input_data.get("tool_name", "")
    if not tool_name.startswith("mcp__linear__"):
        return {}  # Only handle Linear MCP failures

    key = tool_use_id or tool_name
    retry_count = _retry_counts.get(key, 0)

    if retry_count >= MAX_RETRIES:
        # Surface to operator — do not silently swallow
        _retry_counts.pop(key, None)
        return {
            "systemMessage": (
                f"Linear tool {tool_name} failed after {MAX_RETRIES} retries. "
                "Do not retry. Return status='failed' with error_type='tool_failure'."
            )
        }

    # Exponential backoff: 1s, 2s, 4s
    delay = BASE_DELAY_SECONDS * (2 ** retry_count)
    _retry_counts[key] = retry_count + 1

    await asyncio.sleep(delay)

    return {
        "systemMessage": (
            f"Linear tool {tool_name} failed (attempt {retry_count + 1}/{MAX_RETRIES}). "
            f"Waited {delay:.0f}s. Retry the same tool call now."
        )
    }


async def linear_audit_hook(input_data: dict, tool_use_id: str | None, context) -> dict:
    """PostToolUse hook: logs every successful mcp__linear__* call for audit trail."""
    tool_name = input_data.get("tool_name", "")
    if not tool_name.startswith("mcp__linear__"):
        return {}

    # Clear retry counter on success
    if tool_use_id:
        _retry_counts.pop(tool_use_id, None)

    # Async log — fire and forget, does not block agent
    asyncio.create_task(_write_audit_log(tool_name, input_data))
    return {"async_": True, "asyncTimeout": 5000}


async def _write_audit_log(tool_name: str, input_data: dict) -> None:
    """Write to local audit log (replace with structured logger in production)."""
    import json
    from datetime import datetime, timezone
    entry = {
        "ts": datetime.now(timezone.utc).isoformat(),
        "tool": tool_name,

    }
    with open(".claude/linear_audit.log", "a") as f:
        f.write(json.dumps(entry) + "\n")


# Hook configuration to pass to ClaudeAgentOptions
LINEAR_HOOKS = {
    "PostToolUseFailure": [
        HookMatcher(matcher="^mcp__linear__", hooks=[linear_retry_hook])
    ],
    "PostToolUse": [
        HookMatcher(matcher="^mcp__linear__", hooks=[linear_audit_hook])
```

**Tool Use:**

The confirmed Linear MCP tools available via `mcp__linear__*`:
- `mcp__linear__create_issue` — create task/story/subtask
- `mcp__linear__update_issue` — update status, assignee, priority
- `mcp__linear__get_issue` — read a single issue by ID
- `mcp__linear__list_issues` — list issues (ALWAYS pass filters: `teamId` + `limit`)
- `mcp__linear__list_my_issues` — issues assigned to current user
- `mcp__linear__create_comment` — add comment to an issue
- `mcp__linear__list_projects` — list projects in workspace
- `mcp__linear__get_project` — read a single project
- `mcp__linear__list_teams` — list teams
- `mcp__linear__get_team` — read a single team

Do NOT use: community Linear MCP servers (`jerhadf/linear-mcp-server`, `tacticlaunch/mcp-linear`). The official server launched May 2025 and supersedes all community alternatives.

**State Management:**

State is split across three layers — do not conflate them:

| Layer | What Lives Here | Access Pattern |
|-------|----------------|----------------|
| Linear | Operational state: issue status, assignees, PR links, `updatedAt` | `mcp__linear__get_issue` on every write cycle; treat `updatedAt` as optimistic lock |
| `.claude/session_cache.json` | Session IDs for resumption across CLI invocations | Read at CLI start, write after `SystemMessage(subtype="init")` captures `session_id` |
| `knowledge/` directory | Durable patterns, QA insights, architecture decisions | Glob/Grep at cycle start; never write during an active agent session |

The Linear Agent itself is stateless per invocation. All durable state lives in Linear. The agent receives its full input contract on every call and writes its full output contract on exit.

**Context Window Strategy:**

The Linear Agent runs in isolated `query()` calls (not a long-running session). This is the correct pattern for Phase 1 — each CLI command is a fresh invocation. Key constraints:

- Pass the full issue context (title, description, acceptance criteria, `updatedAt`) as structured input in the prompt on every call. Do not rely on session history.
- Never call `mcp__linear__list_issues` without `teamId` + `limit` filters. An unfiltered list on a large workspace can return thousands of tokens of issue data, exhausting the context budget before the agent can act.
- Monitor context budget via the `PreCompact` hook: when the hook fires, log a `context_budget_warning` in the result envelope. This is the signal that the current session is approaching compaction.
- If a `PreCompact` hook fires during a batch operation, abort the current call and restart with a narrower prompt scope.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### 4b.1 Structured Outputs with Pydantic

The Claude Agent SDK does not have a built-in `with_structured_output()` equivalent. The pattern for the Linear Agent is: (1) parse the agent's result text against a Pydantic model in the calling layer, (2) retry the query with an error injection if validation fails, (3) surface after 3 retries.

**Contract models for the Linear Agent (mirror AGENT-CONTRACTS.md exactly):**

```python
"""
src/hsb/contracts/linear.py

Pydantic v2 contract models for the Linear System of Record Agent.
These MUST exactly mirror the JSON schemas in agents/AGENT-CONTRACTS.md.
Do not add or remove fields without updating AGENT-CONTRACTS.md first.
"""
from __future__ import annotations
from enum import Enum
from typing import Literal
from pydantic import BaseModel, Field, model_validator


class LinearOperation(str, Enum):
    create = "create"
    update = "update"
    read = "read"
    link = "link"
    comment = "comment"
    create_subtasks = "create_subtasks"


class LinearInput(BaseModel):
    """Input contract for the Linear System of Record Agent."""
    operation: LinearOperation
    payload: dict  # Operation-specific payload; validated by operation-specific models below

    model_config = {"extra": "forbid"}  # Fail loudly on unknown fields


class LinearEntity(BaseModel):
    """A Linear entity returned after a create/update operation."""
    id: str = Field(..., pattern=r"^LIN-\d+$")
    type: Literal["epic", "user_story", "task", "subtask"]
    url: str = Field(..., pattern=r"^https://linear\.app/")


class LinearOutput(BaseModel):
    """Output contract for the Linear System of Record Agent."""
    operation: str
    result: Literal["success", "failed"]
    linear_entities: list[LinearEntity] = Field(default_factory=list)
    error: str | None = None

    @model_validator(mode="after")
    def failed_must_have_error(self) -> LinearOutput:
        if self.result == "failed" and not self.error:
            raise ValueError("failed result must include error message")
        return self

    model_config = {"extra": "forbid"}


class RuntimeEnvelope(BaseModel):
    """Standard envelope wrapping every agent invocation result."""
    execution_id: str  # UUID
    requested_by: Literal["global_orchestrator", "work_item_orchestrator", "human"]
    skill: str
    agent: str
    input: dict
    output: LinearOutput | None = None
    status: Literal["success", "failed", "blocked"]
    errors: list[str] = Field(default_factory=list)
    next_recommended_action: str | None = None

    model_config = {"extra": "forbid"}
```

**Validation and retry pattern in the calling layer:**

```python
"""
src/hsb/agents/linear_agent.py (validation layer)

Parse agent result text → validate against LinearOutput → retry on failure.
The agent returns JSON in its result text; we parse and validate here,
not inside the agent's context.
"""
import json
import logging
from pydantic import ValidationError
from hsb.contracts.linear import LinearOutput

logger = logging.getLogger(__name__)

MAX_VALIDATION_RETRIES = 3


async def run_validated_linear_agent(
    operation: str,
    payload: dict,
) -> LinearOutput:
    """
    Run the Linear Agent and validate the result against LinearOutput.
    Retries up to MAX_VALIDATION_RETRIES times on validation failure.
    On final failure, raises ValueError with the last validation error.
    """
    from hsb.agents.linear_agent import run_linear_agent

    prompt = (
        f"Execute Linear operation '{operation}' with this payload:\n"
        f"```json\n{json.dumps(payload, indent=2)}\n```\n\n"
        "Return your result as a JSON object matching this schema:\n"
        "{ operation, result: 'success'|'failed', linear_entities: [...], error?: string }"
    )

    last_error: ValidationError | None = None

    for attempt in range(1, MAX_VALIDATION_RETRIES + 1):
        result_text = await run_linear_agent(prompt)

        if result_text is None:
            logger.warning("Attempt %d: agent returned None result", attempt)
            continue

        # Extract JSON from result text (agent may wrap in markdown)
        try:
            json_start = result_text.index("{")
            json_end = result_text.rindex("}") + 1
            raw = json.loads(result_text[json_start:json_end])
        except (ValueError, json.JSONDecodeError) as e:
            logger.warning("Attempt %d: could not parse JSON from result: %s", attempt, e)
            last_error = e  # type: ignore
            # Inject correction into next attempt via modified prompt
            prompt += f"\n\nPrevious attempt returned invalid JSON: {e}. Return ONLY valid JSON."
            continue

        try:
            output = LinearOutput.model_validate(raw)
            logger.info("Attempt %d: validation succeeded", attempt)
            return output
        except ValidationError as e:
            last_error = e
            logger.warning(
                "Attempt %d: LinearOutput validation failed:\n%s",
                attempt,
                e.json(indent=2),
            )
            # Inject the exact validation errors so the agent can self-correct
            prompt += (
                f"\n\nPrevious attempt returned invalid output. Validation errors:\n"
                f"{e.json(indent=2)}\nFix these errors and return corrected JSON."
            )

    raise ValueError(
        f"Linear Agent failed validation after {MAX_VALIDATION_RETRIES} attempts. "
        f"Last error: {last_error}"
    )
```

**Retry logic summary:**
- Up to 3 validation retries (distinct from the SDK's internal tool-level retries)
- Each retry injects the exact `ValidationError` details into the next prompt so the agent can self-correct
- After 3 failures: raise `ValueError`, log the full error, surface to operator CLI
- Log at `WARNING` level per attempt; log at `ERROR` on final failure

### 4b.2 Async-First Design

**How async works in the Claude Agent SDK:**

The `query()` function returns an `AsyncGenerator`. The agent loop, tool execution, and all I/O are async. The correct pattern is always `async for message in query(...)`. The SDK internally uses `asyncio` event loops and is not thread-safe — do not share `ClaudeAgentOptions` objects across threads.

**The one common mistake — `asyncio.run()` inside a running event loop:**

```python
# WRONG: Typer callback that is itself called from an async context
@app.command()
def create_issue():
    # If the calling context already has a running event loop, this raises:
    # RuntimeError: This event loop is already running
    asyncio.run(run_linear_agent("..."))

# CORRECT: Typer is synchronous at the CLI boundary — asyncio.run() is safe here
# BUT: never nest asyncio.run() inside another coroutine or async function
@app.command()
def create_issue():
    result = asyncio.run(run_linear_agent("..."))  # Safe: Typer is sync
    print(result)

# CORRECT: Inside async code, await or async for — never asyncio.run()
async def orchestrate():
    result = await run_linear_agent("...")  # Not asyncio.run()
```

**Stream vs. await — when to use each:**

| Pattern | Use For | Example |
|---------|---------|---------|
| `async for message in query(...)` | Streaming — show operator progress in real time; long-running tasks | Linear Agent CLI output, any user-facing operation |
| `await collect_result(query(...))` | Single-turn result collection — structured output validation; CI/CD pipelines | Pydantic validation layer; `run_validated_linear_agent()` |

For structured output validation (Section 4b.1), collect the full `ResultMessage` before attempting `model_validate()`. Do not attempt validation mid-stream.

**Parallel dispatch in Phase 2+:**

```python
# asyncio.gather() for parallel Work Item Orchestrators
results = await asyncio.gather(
    *[run_linear_agent(item_prompt) for item_prompt in ready_items],
    return_exceptions=True,  # Catch individual failures without stopping the batch
)
```

### 4b.3 Prompt Engineering Discipline

**System vs. user prompt separation:**

The `system_prompt` field in `ClaudeAgentOptions` sets the agent's behavioral contract. It is injected before every conversation turn and persists across context compaction. Use it for:
- Role definition ("You are the Linear Agent for HSBTech")
- Tool use rules ("Always validate before calling mcp__linear__* tools")
- Retry instructions ("On tool failure, retry up to 3 times with 1s/2s/4s backoff")
- Output format requirements ("Return results as JSON matching LinearOutput schema")

The `prompt` argument to `query()` is the user prompt — the specific task for this invocation. Use it for:
- The operation type and payload
- The current Linear state context (injected fresh each call)
- Any cycle-specific context (Linear issue ID, `updatedAt` value for optimistic lock)

**Never combine behavioral rules into the user prompt.** Rules in the user prompt are not guaranteed to survive context compaction.

**Few-shot guidance:**

Do not use inline few-shot examples in the system prompt for the Linear Agent. The contract schema is precise enough that the agent can follow it from the schema alone. If few-shot is needed (e.g., for complex `create_subtasks` payloads), use dynamic injection: read an example from `knowledge/patterns/` and append it to the user prompt for that specific operation.

**Token budget:**

```python
ClaudeAgentOptions(

    # 20 turns = sufficient for read → validate → write → verify.
    # Do NOT leave unbounded (default is no limit — dangerous in production).
    max_turns=20,

    # system_prompt: keep under 2,000 tokens. Verbose system prompts consume
    # context budget that the agent needs for tool results.
    system_prompt="...",  # Benchmark: < 500 tokens for Phase 1 Linear Agent
)
```

### 4b.4 Context Window Management

The Linear Agent runs stateless per invocation — each `query()` call is a fresh context. This is intentional and is the correct pattern for Phase 1. The following applies when sessions grow long (batch operations, Phase 2+ orchestration sessions):

**Automatic compaction (SDK built-in):**

The SDK automatically compacts conversation history when the context budget is exceeded. The `PreCompact` hook fires before compaction. Use it to:
1. Archive the full transcript to a local file before it is summarized
2. Emit a `context_budget_warning` in the result envelope
3. If the compaction would occur mid-write-operation, abort and restart with a narrower scope

```python
async def pre_compact_handler(input_data: dict, tool_use_id: str | None, context) -> dict:
    """Archive transcript and warn before SDK auto-compaction."""
    import json
    from datetime import datetime, timezone
    transcript_path = input_data.get("transcript_path")
    ts = datetime.now(timezone.utc).strftime("%Y%m%dT%H%M%S")
    archive_path = f".claude/compaction_archive_{ts}.jsonl"
    if transcript_path:
        import shutil
        shutil.copy(transcript_path, archive_path)
    # Inject warning into conversation so the agent knows context was compacted
    return {
        "systemMessage": (
            "CONTEXT COMPACTION TRIGGERED. "
            "Re-read the current Linear issue state before proceeding. "
            "Do not assume previously-read data is still accurate."
        )
    }

LINEAR_HOOKS["PreCompact"] = [HookMatcher(hooks=[pre_compact_handler])]
```

**Preventing overflow on Linear list operations:**

Never call `mcp__linear__list_issues` without filters. Enforce this at the system prompt level AND via a `PreToolUse` hook:

```python
async def enforce_list_filters(input_data: dict, tool_use_id: str | None, context) -> dict:
    """Block un-filtered list_issues calls that would overflow context."""
    if input_data.get("tool_name") != "mcp__linear__list_issues":
        return {}
    tool_input = input_data.get("tool_input", {})
    if not tool_input.get("teamId") and not tool_input.get("projectId"):
        return {
            "hookSpecificOutput": {
                "hookEventName": "PreToolUse",
                "permissionDecision": "deny",
                "permissionDecisionReason": (
                    "list_issues requires teamId or projectId filter to prevent "
                    "context overflow. Add a filter and retry."
                ),
            }
        }
    return {}

LINEAR_HOOKS["PreToolUse"] = [
    HookMatcher(matcher="mcp__linear__list_issues", hooks=[enforce_list_filters])
]
```

**For Phase 2+ multi-session orchestration:**

Each Work Item Orchestrator must be invoked with `context: fork` (subagent mode via `AgentDefinition`) so it runs in an isolated context rather than accumulating state in one long session. The full Linear issue (description, acceptance criteria, `updatedAt`) must be re-injected at every cycle start — never rely on conversation history to carry requirements across cycles.

### 4b.5 Cost and Latency Budget

**Per-call cost estimate (claude-sonnet-4-6, 2026 pricing):**

A typical Phase 1 Linear Agent call for a single `create_issue` operation:
- Input tokens: ~2,000 (system prompt + user prompt + Linear MCP tool definitions)
- Output tokens: ~500 (tool calls + result)
- Estimated cost: ~$0.01–$0.03 per operation at Sonnet pricing
- At 100 operations/day: ~$1–3/day; at 1,000 operations/day: ~$10–30/day

Tool definitions for `mcp__linear__*` are injected into context on every call. With tool search enabled (SDK default), only the tools relevant to the current turn are loaded. Verify tool search is not disabled: do not pass `tool_search=False` unless you have a specific reason.

**Exact-match caching:**

The SDK does not provide built-in result caching. For read-heavy operations (`get_issue`, `list_projects`):

```python
import functools
import time
from typing import Any

# Simple TTL cache for Linear read operations (not for writes)
_cache: dict[str, tuple[float, Any]] = {}
CACHE_TTL_SECONDS = 30  # Linear state changes slowly in dev workflows


def cached_linear_read(operation_key: str, ttl: int = CACHE_TTL_SECONDS):
    """Decorator for caching Linear read results. Do NOT use for writes."""
    def decorator(fn):
        @functools.wraps(fn)
        async def wrapper(*args, **kwargs):
            now = time.time()
            if operation_key in _cache:
                cached_at, value = _cache[operation_key]
                if now - cached_at < ttl:
                    return value
            result = await fn(*args, **kwargs)
            _cache[operation_key] = (now, result)
            return result
        return wrapper
    return decorator
```

**Cheaper models for sub-tasks:**

Not applicable in Phase 1 — all operations go through the Linear Agent with `claude-sonnet-4-6`. In Phase 2+:

| Sub-task | Recommended Model | Rationale |
|----------|------------------|-----------|
| Issue status classification | `claude-haiku-3-5` | Simple enum routing; no tool use |
| Duplicate issue detection | `claude-haiku-3-5` | Semantic similarity only |
| Linear state summarisation | `claude-haiku-3-5` | Summarise large list_issues output before injecting into Sonnet context |
| Implementation / QA / PR review | `claude-sonnet-4-6` | Complex multi-step tool use |

Route cheap classification calls through `claude-haiku-3-5` before passing the result to the main Sonnet agent loop. This reduces the input token count for the expensive call and cuts latency on the classification step.

## 6. Guardrails

> This is an internal CLI with no real-time user-facing surface. Safety guardrails focus on data integrity and Linear state correctness, not content moderation.

### Online (Real-Time)

These run on every agent invocation. They are implemented as Claude Agent SDK hooks and fire synchronously before or during tool execution.

| Guardrail | Hook | Trigger | Intervention |
|-----------|------|---------|--------------|
| **Pydantic contract enforcement** | Calling layer (pre-agent) | `LinearInput.model_validate()` raises `ValidationError` on the input payload before `query()` is called | Block — raise `ValueError` with the full `ValidationError` detail; do not invoke the agent; surface to operator CLI immediately |
| **MCP tool prefix verification** | `PostToolUse` (inline check in `linear_audit_hook`) | Any captured tool call name does not match `^mcp__linear__` | Flag — log `WARN: unexpected tool prefix {name}` to audit log and CLI; do not block (the call already completed) but mark the result envelope with `prefix_mismatch: true` for operator inspection |
| **Retry cap enforcement** | `PostToolUseFailure` (`linear_retry_hook`) | `_retry_counts[key] >= MAX_RETRIES (3)` | Block further retries — inject `systemMessage` instructing agent to return `result: "failed"`; clear retry counter; surface `error_type: "tool_failure"` to operator |
| **Unfiltered `list_issues` block** | `PreToolUse` (`enforce_list_filters`) | `mcp__linear__list_issues` called without `teamId` or `projectId` in `tool_input` | Deny — return `permissionDecision: "deny"` with reason; forces agent to add a filter before retrying the tool call |

### Offline (Flywheel)

These run as batch checks over the `.claude/linear_audit.log` after operations complete. They are not latency-sensitive and feed the improvement loop.

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| **Linear write success rate** | All write operations (`create_issue`, `update_issue`) in the audit log; computed daily from log entries | If success rate drops below 95% over a 24-hour window: operator reviews audit log for pattern; check for Linear API degradation or schema drift |
| **Retry rate per operation** | All `PostToolUseFailure` entries in audit log; count retries per operation type; compute weekly average | If retry rate for any operation type exceeds 10% over a weekly window: investigate Linear API reliability or hook timing; consider increasing backoff ceiling |
| **Pydantic validation failure rate** | Count of `ValidationError` raised in `run_validated_linear_agent()` before final success; logged at `WARNING` level | If validation failure rate (retries needed before success) exceeds 5% of invocations: review system prompt output format instructions; check for model version drift changing JSON formatting behavior |
| **`updatedAt` atomicity check failures** | Audit log entries where `post_updated_at <= pre_updated_at`; computed on every batch run | Any occurrence triggers immediate operator review — this indicates either a cache hit on the pre-write read or a concurrent mutation; treat as data integrity incident |
| **Schema drift signal** | Weekly: feed a sample of 5 recent `LinearOutput` payloads from audit log through `LinearOutput.model_validate()` with a snapshot of the current AGENT-CONTRACTS.md schema | Any `ValidationError` on a production payload that previously passed: open schema drift ticket; freeze deployments until contracts are re-synchronized |

## 7. Production Monitoring

> Lightweight configuration appropriate for pre-production internal tooling. All monitoring is local — no external platform account required.

**Tracing Tool:** Arize Phoenix (open-source, self-hosted at `http://localhost:6006`)

Arize Phoenix is confirmed as the default — no existing tracing tool was detected in the repository, and Phoenix's framework-agnostic OpenTelemetry instrumentation is the correct choice for a Claude Agent SDK project (LangSmith override does not apply here). Phoenix traces every `query()` invocation, captures tool call sequences, and surfaces retry patterns visually.

**Initialization (one-time, run before agent sessions during development):**

```python
# src/hsb/observability.py
import phoenix as px
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter

def init_tracing(project_name: str = "hsb-linear-agent") -> None:
    px.launch_app()  # http://localhost:6006
    provider = TracerProvider()
    exporter = OTLPSpanExporter(endpoint="http://localhost:6006/v1/traces")
    provider.add_span_processor(BatchSpanProcessor(exporter))
    trace.set_tracer_provider(provider)
```

**Key Metrics to Track:**

| Metric | Source | How Computed |
|--------|--------|--------------|
| **Linear write success rate** | `.claude/linear_audit.log` | `count(result="success") / count(all writes)` — daily |
| **Retry rate** | `.claude/linear_audit.log` (`PostToolUseFailure` entries) | `count(retry_attempts > 0) / count(all tool calls)` — weekly |
| **Pydantic validation failure rate** | Application `WARNING` logs from `run_validated_linear_agent` | `count(validation_retries_needed) / count(all invocations)` — weekly |
| **`updatedAt` atomicity violations** | `.claude/linear_audit.log` | `count(post_updated_at <= pre_updated_at)` — every batch run; any non-zero value is an incident |
| **Agent turn consumption** | Phoenix traces (turn count per `query()` span) | Flag any invocation consuming > 15 turns (approaching the `max_turns=20` cap) |

**Alert Thresholds:**

| Threshold | Action |
|-----------|--------|
| Linear write success rate < 95% in any 24-hour window | Operator reviews audit log; check Linear API status; do not trigger new agent runs until investigated |
| Any `updatedAt` atomicity violation (i.e., count > 0) | Immediate operator review — treat as data integrity incident; inspect for cache misuse or concurrent mutations |
| Validation failure rate > 5% of invocations in a week | Review system prompt JSON output instructions; check if model version was updated; compare against reference dataset |
| Any agent invocation consuming ≥ 18 turns out of 20 | Log `WARN: approaching max_turns`; inspect the specific trace in Phoenix for loop or retry storm patterns |
| Retry rate > 10% for any single operation type over a week | Investigate Linear API reliability for that operation; consider raising `BASE_DELAY_SECONDS` |

**Smart Sampling Strategy:**

This is an internal tool run by a small team (HSBTech engineers). Full audit log review is feasible at low volume. Apply signal-based filtering as volume grows:

1. **Always review:** every `updatedAt` atomicity violation; every invocation that hit `max_turns=20`; every `result: "failed"` outcome.
2. **Weekly sample:** 3 randomly selected successful write operations — spot-check audit log entries for correct `pre_updated_at` / `post_updated_at` pairs and non-empty `linear_entities`.
3. **On retry spike:** when weekly retry rate exceeds 10% for any operation type, pull all retry-involved traces from Phoenix for that operation and inspect tool call sequences.
4. **Post schema change:** after any AGENT-CONTRACTS.md update, run the full reference dataset and manually inspect all 12 scenario outputs before resuming normal operation.
