## 1. System Classification

**System Type:** Hybrid — Multi-Agent (UAT/Intelligence/Risk add to existing WIO + Global Orchestrator) + Autonomous Task (UAT validation, Risk LLM trigger detection) + Lightweight Retrieval (Knowledge Store via Glob+Grep, no vector DB).

**Description:**
Phase 5 adds three "enhancement" agents on top of the validated Phase 1–4 delivery loop in the HSBTech (HSB) task management system. (1) The **Intelligence Agent** is embedded inline into the WIO system prompt (skills 10+11) — it queries the local file-based Knowledge Store before Builder execution to enrich the work item with relevant prior knowledge, then evaluates QA findings and implementation notes after each cycle and writes new Knowledge Store entries when warranted. (2) The **UAT Agent** is a Claude Agent SDK session (skill 08 inline) auto-dispatched by the Global Orchestrator when all child tasks of a User Story reach `qa_status = approved`; it validates the User Story against acceptance criteria from a user-acceptance perspective (no code review, no PR creation) and produces scenario pass/fail results that drive a standard fix-task loop. (3) The **Risk Agent** is a pure Python class (skills 12+13) that produces deterministic quality scores and a prioritized work-item queue consumed by the Global Orchestrator; it uses a single isolated Claude Agent SDK invocation only for skill 14 (Auto-Improvement Trigger detection) and never writes to Linear without explicit delegation through the Linear Agent. Users are the developer-operator (the human running the HSB CLI to manage their personal/team Linear backlog) and indirectly the Linear team (whose work items the system reads and writes). "Good" looks like: enrichment that demonstrably reuses prior project knowledge to avoid repeated mistakes; UAT verdicts that track what a careful human reviewer would conclude; priority orderings that systematically defer high-risk items and surface high-value low-risk items; and zero unauthorized Linear mutations from the Risk Agent.

**Critical Failure Modes:**
1. **UAT false-pass** — UAT Agent marks a User Story as "approved" when an acceptance criterion is actually unmet, causing broken work to flow downstream (UATA-02). Stakes: a defective User Story is closed and the team trusts the queue.
2. **Risk Agent unauthorized Linear write** — Risk Agent's skill 14 LLM invocation creates Linear work items or comments without explicit delegation through the Linear Agent (RISK-04 violation). Stakes: silent Linear pollution that erodes operator trust in the system.
3. **Knowledge Store contamination** — Intelligence Agent persists low-signal, duplicated, or incorrect entries (e.g., one-off bugs presented as patterns) that pollute future enrichment retrieval and propagate bad guidance (INTL-02 ingestion criteria violated).
4. **Sub-subagent dispatch leak** — Any Phase 5 SDK session spawns a nested SDK session (forbidden by RUNTIME-EXECUTION.md golden rule and reinforced in WORC-02). Stakes: uncontrolled token usage and process tree, breaks the audit trail.
5. **Context exhaustion in WIO** — Inline injection of skills 10+11 alongside skills 02+03+04+05+06 pushes WIO over the validated context budget from Phase 3, causing truncation of work item context or QA results mid-cycle (Pitfall 5 in `.planning/research/PITFALLS.md`).
6. **QA cycle cap bypass via UAT fix loop** — UAT-driven fix subtasks form a loop that exceeds the project-wide max_qa_cycles=3 invariant when the parent re-triggers UAT after fixes are merged (Pitfall 2: QA runaway).

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer tooling / AI-augmented software delivery (three overlapping sub-domains: agile UAT practice, engineering knowledge management, and risk-based backlog prioritization)

**User Population:** Developer-operator: a solo developer or small team (2–5 people) running HSB CLI commands to manage their Linear backlog. The operator has engineering judgment but limited bandwidth — they rely on agent verdicts to decide what to act on next. They are the final authority on every merge and every Linear state transition.

**Stakes Level:** High (not regulated, but trust-dependent — the system's autonomy thesis collapses if any one of the three agents produces outputs the operator cannot rely on without re-checking manually)

**Output Consequence:** UAT Agent verdicts close User Stories and release backlog capacity; a false-pass means broken work is declared done and resurfaces as a production bug or a confused downstream dependency. Intelligence Agent entries persist permanently in the Knowledge Store and are injected into future work items; low-signal entries degrade every future Builder enrichment until the store is manually audited. Risk Agent priority queues determine which work the operator picks up next; a systematically wrong ordering causes the operator to do low-value or high-risk work first, compounding delivery risk over time.

### What Domain Experts Evaluate Against

**Sub-domain 1: User Acceptance Testing**

```
Dimension: Acceptance-criteria coverage completeness
Good (expert accepts): Every acceptance criterion listed on the User Story has a corresponding
  UAT scenario with an explicit expected outcome and a pass or fail verdict. No criterion is
  silently skipped, even if it appears untestable from a static review.
Bad (expert flags): One or more acceptance criteria appear in the User Story but have no
  matching UAT scenario in the report. The UAT verdict is "approved" despite the gap.
Stakes: Critical
Source: ISTQB Agile Testing Extension body of knowledge; AgileUAT framework (Haugset &
  Hanssen, ResearchGate 2015); BABOK v3 acceptance criteria practices — a UAT sign-off
  without full-criteria coverage is the primary cause of false-pass defects reaching production.
```

```
Dimension: Scenario specificity vs. test-criteria mismatch
Good (expert accepts): Each UAT scenario states the exact user action, the system state
  precondition, and the observable outcome that constitutes a pass — in language matching
  the acceptance criterion it covers. Pass/fail determination is explained by comparing
  expected vs. actual behavior, not by a general impression.
Bad (expert flags): UAT scenario descriptions are paraphrased from the acceptance criterion
  without specifying the observable outcome. The agent marks a scenario "pass" without
  recording what actual behavior was observed and how it was compared to expected.
Stakes: Critical
Source: Crispin & Gregory "Agile Testing" (2009) — acceptance tests are only meaningful
  when expected and actual are independently statable; arxiv.org/abs/2504.07244 (AutoUAT
  industrial case study, 2025) found 24% of LLM-generated test scripts required full
  regeneration due to insufficient context specificity.
```

```
Dimension: Scope boundary enforcement
Good (expert accepts): UAT Agent evaluates whether the User Story's feature behaves
  correctly from the user's perspective. It does not comment on code quality, linting,
  architecture, or anything outside the acceptance criteria.
Bad (expert flags): UAT report includes findings about implementation quality ("the
  function is inefficient"), style ("variable names are unclear"), or scope outside the
  User Story ("this doesn't handle a future edge case not in the criteria").
Stakes: High
Source: UATA-04 in REQUIREMENTS.md; ISTQB distinction between code review (structural)
  and acceptance testing (behavioral/business). Scope creep in UAT verdicts erodes
  operator trust by making UAT indistinguishable from a second QA pass.
```

**Sub-domain 2: Engineering Knowledge Management**

```
Dimension: Knowledge entry signal threshold (recurring pattern vs. one-off noise)
Good (expert accepts): A knowledge entry records a finding that: (a) applies to a class
  of future tasks identifiable by technology, pattern, or domain (not just the specific
  issue that produced it), AND (b) provides a concrete recommendation that changes how
  a future Builder would approach the work. The entry cites the Linear issue and PR as
  evidence.
Bad (expert flags): An entry records a single-occurrence bug fix with no indication it
  will recur ("fixed null pointer in method X"). An entry states an observation without
  a recommendation ("QA found an error"). An entry duplicates an existing entry with
  different wording.
Stakes: High
Source: Google SRE Postmortem Culture (sre.google) — the criterion for a postmortem
  action item is that it prevents a recurrence, not that it documents an occurrence.
  ADR practice (adr.github.io) — architectural decisions are recorded only when they
  affect future decisions, not as a log of every choice made.
```

```
Dimension: Entry reusability and applicability scope
Good (expert accepts): The entry's "applicability" field explicitly identifies the
  conditions under which a future Builder should retrieve and apply it — by task type,
  technology stack component, or acceptance-criteria pattern. A developer reading the
  entry in isolation can determine in under 30 seconds whether it applies to their
  current task.
Bad (expert flags): The applicability field is absent, says "all tasks," or is too
  narrow ("only for PR #47"). The entry requires reading the original Linear issue to
  understand its scope. A developer retrieves the entry and finds it irrelevant to
  their context more than 50% of the time.
Stakes: High
Source: Atlassian Knowledge Management research (2024 KM Trends, enterprise-knowledge.com)
  — knowledge reuse fails primarily due to poor applicability metadata, not missing
  content. "Don't Make Me Think" principle (Krug, 2014) applied to engineering docs:
  if the reader must reconstruct context, the entry has failed.
```

**Sub-domain 3: Risk-Based Prioritization**

```
Dimension: Risk score traceability (operator can audit the score)
Good (expert accepts): The priority queue entry for each work item shows the quality
  score with its component penalties (e.g., "-10 for 2 QA failures, -5 for 1 fix
  subtask") and the resulting risk level, so the operator can verify the score without
  querying Linear themselves. The score changes only when the underlying QA/UAT history
  changes.
Bad (expert flags): The score appears as a single number with no breakdown. Two items
  with the same score appear in an arbitrary order with no tiebreaker rationale.
  The score is the same across consecutive cycles despite new QA findings having been
  recorded since the last run.
Stakes: High
Source: WSJF (SAFe framework, scaledagile.com) and RICE scoring practitioners document
  that scores are dismissed by senior engineers when the inputs are opaque — "a number
  without a story is not trusted" (Fygurs prioritization frameworks comparison, 2024).
  ICE scoring (Sean Ellis, growth hacking literature) — Confidence component exists
  specifically because scores without uncertainty acknowledgment are overfit to available data.
```

### Known Failure Modes in This Domain

**1. UAT false-pass from acceptance-criteria gap (most critical)**
The UAT Agent approves a User Story while one or more acceptance criteria have no corresponding scenario in the report. This happens when acceptance criteria are implicit (written as narrative rather than testable conditions), ambiguous, or span multiple behaviors. In production deployments of LLM-based test generation, 8% of generated scenarios were discarded entirely due to fundamental incompatibility with validation (arxiv.org/abs/2504.07244). An automated agent with no human override is especially vulnerable because there is no fallback reviewer to catch the gap.

**2. Knowledge Store graveyard (compounding noise)**
The Knowledge Store accumulates entries that pass the ingestion criteria at the time of writing but become stale as the codebase evolves. Entries that recorded a workaround for a specific library version remain in the store long after the library is upgraded. Entries that captured a one-off architectural decision that was later reversed continue to surface in enrichment. Unlike a Confluence wiki, which decays visibly (users notice when a page hasn't been edited in two years), a flat markdown store has no staleness signal. The Intelligence Agent will retrieve and inject stale entries without any mechanism to know they are outdated.

**3. Risk score gaming / operator dismissal**
If the Risk Agent's quality score formula produces counterintuitive rankings — for example, giving a high-priority score to a work item the operator knows is already de-risked, or systematically demoting items they intend to work on next — the operator will start ignoring the priority queue and revert to manual ordering. Once the operator stops trusting the queue, the Risk Agent provides zero value despite running on every cycle. This failure mode is invisible in logs (the operator simply doesn't follow the recommendation) and is detected only by observing behavior over time.

**4. Scope bleed from UAT into QA territory**
UAT Agent is scoped to user-acceptance validation (UATA-04), not code review. When the agent's reasoning is driven by implementation details visible in the PR diff (which it should not be accessing), it produces findings that duplicate or contradict the QA Agent's findings. This creates two conflicting authority sources for the same work item, and the operator must adjudicate. In practice, operators interpret scope bleed as the system "not understanding what it's supposed to do," which accelerates trust erosion.

### Regulatory / Compliance Context

None identified for this deployment context. The HSB system is internal developer tooling with no patient data, financial transaction data, or personally identifiable end-user data in scope. Linear workspace data (issue titles, descriptions, comments) is handled under the operator's existing Linear subscription terms.

The relevant non-regulatory constraint is **operator autonomy preservation**: no agent may take a consequential irreversible action (create, update, or delete Linear work items; create PRs; merge code) without explicit delegation from the operator or through the designated intermediary agent (Linear Agent). This is an architectural constraint enforced by RISK-04, INTL-04, and the WORC-02 golden rule — not a regulatory requirement, but it carries equivalent weight for system trustworthiness.

## 2. Framework Decision

**Selected Framework:** Claude Agent SDK (`claude-agent-sdk` Python package) — extends the framework already in use across Phases 1–4. Optional supplemental retrieval utility: `rank-bm25` (pure Python BM25 library) — *deferred unless Glob+Grep retrieval proves insufficient at MVP scale*.

**Version:** `claude-agent-sdk >= 0.1.73, < 0.2.0` (matches the version pinned in Phases 1–4 per `.planning/research/STACK.md`). `rank-bm25 ^0.2.2` if introduced.

**Rationale:**
The project committed to Claude Agent SDK in Phase 1 and `.planning/REQUIREMENTS.md` explicitly lists LangChain, LangGraph, and CrewAI in "Out of Scope." All three Phase 5 agents map onto SDK patterns already proven in earlier phases without introducing new primitives:

- **UAT Agent** uses the same Claude Agent SDK session + inline skill injection pattern established by WIO in Phase 3 (Decision D-03 in `05-CONTEXT.md`). Skill 08 content is concatenated into the system prompt; the session interacts only with the Linear Agent and shell tools — no nested SDK sessions.
- **Intelligence Agent** does not introduce a new agent process. Skills 10+11 are appended to the WIO system prompt's existing skill injection set (skills 02+03+04+05+06), per Decision D-04. This consolidates Intelligence behavior into the WIO's single Agent SDK session and respects the WORC-02 "no sub-subagent dispatch" golden rule.
- **Risk Agent** is a pure Python class (Decision D-08) — quality scoring (skill 12) and adaptive prioritization (skill 13) are deterministic math, not LLM tasks, so no framework abstraction is needed. The single LLM call for skill 14 (Auto-Improvement Trigger detection, Decision D-09) reuses the same `claude-agent-sdk` `query()` pattern as WIO; the resulting suggestions are returned to the Global Orchestrator and only persisted to Linear when explicitly delegated through the Linear Agent (RISK-04).

The Knowledge Store retrieval (INTL-01) uses Glob+Grep for MVP per `.planning/research/STACK.md` — semantic search is explicitly deferred to v2 (ADVL-01). No vector DB, no LlamaIndex, no embedding model. If retrieval precision degrades as the Knowledge Store grows past ~50 entries, `rank-bm25` is the minimum-friction upgrade — it adds scored full-text ranking with zero infrastructure cost and no new framework dependency.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain | Explicitly listed in REQUIREMENTS.md "Out of Scope". Adds abstraction layer over the same Anthropic API the SDK already wraps; ecosystem advantage is irrelevant since Phase 5 only integrates Linear + GitHub + filesystem (already wired in Phases 1–2). |
| LangGraph | Explicitly excluded by REQUIREMENTS.md. Graph-state machine value would compete with the project's intentional Pure-Python orchestrator pattern (Phase 4 Global Orchestrator, Phase 3 WIO state machine). Adopting it now would invalidate built-and-tested Phase 1–4 architecture. |
| CrewAI | Explicitly excluded by REQUIREMENTS.md. Multi-agent role abstraction conflicts with WORC-02 (no sub-subagent dispatch) — a CrewAI "crew" is exactly the dispatch pattern the project bans. |
| LlamaIndex | Strong RAG framework, but its value is retrieval primitives (parsers, embedders, vector stores) the project has explicitly deferred (ADVL-01) and forbidden (no vector DB). Glob+Grep is sufficient at MVP scale; rank-bm25 is the cheaper upgrade if needed. |
| OpenAI Agents SDK / Google ADK | Bound to OpenAI / Gemini providers. Project is committed to Claude (memory note + STACK.md). Switching providers would also break the OAuth2-only constraint (no API keys) — both vendors require API keys. |
| AutoGen / AG2 / Microsoft Agent Framework | Conversational multi-agent abstraction that conflicts with WORC-02 and the "one action per WIO cycle" golden rule. Ecosystem fragmentation (AutoGen maintenance mode) is an additional risk for a small-team project. |
| Haystack | NLP-pipeline-first design. Phase 5 is not a pipeline framework concern — it is agent orchestration + a thin retrieval surface. Wrong tool. |

**Vendor Lock-In Accepted:** Yes (Anthropic Claude). Already accepted at Phase 1 — Phase 5 inherits this constraint. Mitigations: model selection is configurable per call (Claude Opus / Sonnet / Haiku based on task), Pydantic contracts isolate LLM output shape from business logic, and the deterministic Risk Agent core (skills 12+13) is provider-independent so the highest-volume Phase 5 work has zero LLM dependency.

### Entry Point Patterns

#### Pattern A — WIO Long-Running Session with Skills 10+11 Injected

Used by the Work Item Orchestrator. `ClaudeSDKClient` maintains context across Enrich → Build → QA → Store steps in a single session. "Agent" is absent from `allowed_tools` — this is the WORC-02 enforcement mechanism.

```python
from pathlib import Path
from claude_agent_sdk import ClaudeSDKClient, ClaudeAgentOptions, AssistantMessage, ResultMessage, TextBlock


def load_skill(path: str) -> str:
    return Path(path).read_text(encoding="utf-8")


async def run_work_item_orchestrator(work_item_id: str, work_item_json: str) -> str:
    # Assemble system prompt: skills 02–06 (existing) + skills 10+11 (Phase 5 additions)
    system_prompt = "\n\n---\n\n".join([
        load_skill(".claude/skills/backlog-planning/SKILL.md"),
        load_skill(".claude/skills/implementation/SKILL.md"),
        load_skill(".claude/skills/qa-review/SKILL.md"),
        load_skill(".claude/skills/git-pr-management/SKILL.md"),
        load_skill(".claude/skills/linear-system-of-record/SKILL.md"),
        load_skill(".claude/skills/knowledge-context-enrichment/SKILL.md"),  # skill 10
        load_skill(".claude/skills/knowledge-storage/SKILL.md"),             # skill 11
    ])

    options = ClaudeAgentOptions(
        system_prompt=system_prompt,
        # "Agent" is deliberately absent — enforces WORC-02 (no sub-subagent dispatch)
        allowed_tools=["Read", "Write", "Edit", "Bash", "Glob", "Grep"],
        mcp_servers={
            "linear": {"command": "npx", "args": ["-y", "mcp-remote", "https://mcp.linear.app/mcp"]}
        },
        permission_mode="acceptEdits",
        max_turns=40,           # Hard ceiling — prevents runaway loops (PITFALLS.md Pitfall 2)
        model="claude-sonnet-4-6",
        cwd=str(Path.cwd()),    # Must be consistent for session resume to work
    )

    result_text = ""
    session_id = None

    async with ClaudeSDKClient(options=options) as client:
        # Step 1: Intelligence enrichment (skill 10) — before Builder
        await client.query(
            f"Enrich work item {work_item_id} from the Knowledge Store before implementation. "
            f"Work item context: {work_item_json}"
        )
        async for msg in client.receive_response():
            if isinstance(msg, AssistantMessage):
                for block in msg.content:
                    if isinstance(block, TextBlock):
                        print(block.text, end="", flush=True)

        # Steps 2–4: Builder → Git → QA lifecycle (existing WIO pattern)
        await client.query(
            f"Execute the full build-git-qa cycle for {work_item_id}. "
            "Apply max_qa_cycles=3. Report final qa_status."
        )
        async for msg in client.receive_response():
            if isinstance(msg, ResultMessage):
                session_id = msg.session_id
                result_text = msg.result or ""
                if msg.stop_reason == "error_max_turns":
                    raise RuntimeError(f"WIO hit max_turns for {work_item_id}")

        # Step 5: Intelligence storage (skill 11) — after QA result
        await client.query(
            "Evaluate QA findings and implementation notes from this cycle. "
            "Apply skill 11 ingestion criteria. Write Knowledge Store entries only for "
            "insights that meet the signal threshold."
        )
        async for msg in client.receive_response():
            pass  # Side-effectful writes to knowledge/ — no return value needed

    return result_text
```

#### Pattern B — Standalone Bounded Session (UAT Agent, skill 08)

One call per User Story. Not resumed. `dontAsk` permission mode — denies any tool not in `allowed_tools`.

```python
async def run_uat_agent(user_story_id: str, acceptance_criteria: list[str]) -> str:
    """Skill 08 embedded as system prompt. No Agent tool = no sub-subagent (WORC-02)."""
    options = ClaudeAgentOptions(
        system_prompt=load_skill(".claude/skills/uat-validation/SKILL.md"),
        allowed_tools=["Read", "Glob", "Grep", "Bash"],  # No "Agent" — WORC-02
        mcp_servers={
            "linear": {"command": "npx", "args": ["-y", "mcp-remote", "https://mcp.linear.app/mcp"]}
        },
        permission_mode="dontAsk",  # Deny anything not in allowed_tools — headless safe
        max_turns=20,
        model="claude-sonnet-4-6",
    )

    scope_block = "\n".join(f"[AC-{i+1}] {c}" for i, c in enumerate(acceptance_criteria))
    prompt = (
        f"Validate User Story {user_story_id}.\n\n"
        "SCOPE BOUNDARY: Only validate the acceptance criteria listed below. "
        "Do not evaluate any feature not explicitly listed. "
        "Any finding without a direct reference to a listed criterion is out of scope.\n\n"
        f"Acceptance criteria:\n{scope_block}\n\n"
        "Return your result as a single JSON object matching the UATResult schema."
    )

    result_text = ""
    async for msg in query(prompt=prompt, options=options):
        if isinstance(msg, ResultMessage):
            result_text = msg.result or ""
            if msg.stop_reason == "error_max_turns":
                raise RuntimeError(f"UAT Agent hit max_turns for {user_story_id}")
    return result_text
```

#### Pattern C — One-Shot Minimal LLM Call (Risk Agent, skill 14 only)

### Common Pitfalls

**Pitfall 1 — Old import path `claude_code_sdk` raises `ModuleNotFoundError`.**
The SDK was renamed in September 2025. Any `from claude_code_sdk import ...` or `from claude_code_sdk import ClaudeCodeOptions` in Phase 1–4 code will fail with the current package. Audit every import in `src/hsb/agents/` before writing Phase 5 code. The correct import root is `claude_agent_sdk`.

**Pitfall 2 — `asyncio.run()` inside an already-running event loop raises `RuntimeError`.**
Parallel Mode runs WIOs inside `asyncio.gather()` — all coroutines share one event loop. Calling `asyncio.run(run_uat_agent(...))` from inside a coroutine raises `RuntimeError: This event loop is already running`. All Phase 5 agent entry points must be `async def` and called with `await`. The only place `asyncio.run()` is legitimate is the top-level CLI entry (`run_loop.py`).

**Pitfall 3 — Session resume silently starts a fresh session when `cwd` differs.**
Sessions are stored under `~/.claude/projects/<encoded-cwd>/`. If `cwd` changes between the original call and the resume call, the SDK cannot find the session file and silently creates a new session — no error is raised. Always pass `cwd=str(Path.cwd())` explicitly in `ClaudeAgentOptions` and assert it is the same value in the session cache before resuming.

**Pitfall 4 — `permission_mode` left as `"default"` in headless mode causes a process hang.**
`"default"` requires a `can_use_tool` callback. Without it, the SDK waits for interactive approval that never arrives. Phase 5 agents in headless execution (CI, parallel mode) must use `"dontAsk"` or `"acceptEdits"`. The hang is silent — no timeout, no error message.

**Pitfall 5 — `max_turns` unset; agent loops indefinitely on repeated tool failures.**
Without `max_turns`, an agent encountering a persistent tool error (e.g., MCP connection drop) loops until context exhaustion or process kill. Always set explicitly: WIO=40, UAT=20, Risk-14=3. Check `ResultMessage.stop_reason == "error_max_turns"` and surface it as a warning requiring human review.

**Pitfall 6 — `ANTHROPIC_API_KEY` in shell environment overrides `CLAUDE_CODE_OAUTH_TOKEN`.**
The SDK checks `ANTHROPIC_API_KEY` at precedence level 3, before `CLAUDE_CODE_OAUTH_TOKEN` at level 5. A developer with a leftover `ANTHROPIC_API_KEY` from another project silently routes all calls through the API key — bypassing the project's OAuth-only constraint. Add a startup assertion: `assert "ANTHROPIC_API_KEY" not in os.environ`.

## 4. Implementation Guidance

### Model Configuration

Phase 5 routes each SDK call to the cheapest model that satisfies the reasoning requirement. Use stable model IDs — not aliases like `"claude-sonnet-latest"` — because aliases can resolve to different versions across Anthropic releases, breaking reproducibility.

| Agent / Call | Model | Rationale |
|---|---|---|
| WIO lifecycle (skills 02–06, 10+11) | `claude-sonnet-4-6` | Full implementation cycle: code generation, QA review, knowledge judgment. Matches Phase 3 validated config. |
| UAT Agent (skill 08) | `claude-sonnet-4-6` | Acceptance-criteria interpretation requires comparable reasoning depth to WIO. |
| Risk Agent skill 14 (AutoImprovement trigger) | `claude-haiku-4-5` | Pattern classification and text generation. Low context, low complexity. Haiku is 5–8x cheaper than Sonnet. |

Key parameters across all Phase 5 calls:
- `max_turns`: WIO=40, UAT=20, Risk-14=3. Hard ceiling on agentic loops.
- `max_budget_usd`: Risk-14=0.05. Hard cost ceiling for the only high-frequency classification call.
- Temperature: not directly exposed in `ClaudeAgentOptions`. The SDK uses the model's default. For Risk-14's one-shot bounded prompt, prompt design substitutes for temperature control.
- Extended thinking (`effort` field): only available on Opus 4.7+. Not used in Phase 5.

### Core Pattern

Three distinct SDK usage patterns, each mapped to one Phase 5 agent:

**`ClaudeSDKClient` (WIO — Pattern A):** The correct primitive for a multi-step lifecycle where each step reads tool results from the previous step. The client automatically continues the session across `client.query()` calls within one `async with` block. Use when: sequential lifecycle steps must share context, Knowledge Store reads from step 1 inform step 2 behavior.

**`query()` standalone (UAT Agent, Risk-14 — Patterns B and C):** Self-contained session per call. The Global Orchestrator passes all required context in the prompt string. No state is carried between invocations at the SDK level. Use when: bounded task, no cross-step context needed, one prompt in one answer out.

**Inline skill injection:** Read `.claude/skills/<name>/SKILL.md` at process start and concatenate into `system_prompt`. This is the established Phase 3 pattern — it gives the SDK full skill content at context initialization rather than relying on progressive-disclosure auto-discovery. Tradeoff: all injected skills count against the system prompt token budget from turn 1, but context pressure is acceptable given Claude Sonnet's 200K token window.

| State Type | Location | Owner | Durability |
|---|---|---|---|
| Work item operational state | Linear (via MCP read/write) | WIO, UAT Agent, Linear Agent | Persistent, authoritative |
| WIO session context (multi-step) | `~/.claude/projects/<cwd>/<session-id>.jsonl` | `ClaudeSDKClient` (automatic) | Persistent across CLI restarts |

| Knowledge Store entries | `knowledge/<category>/<YYYY-MM-DD>-<slug>.md` | Intelligence Agent (skill 11) | Git-tracked, permanent until pruned |

Session ID persistence for WIO resume across CLI restarts:

```python
import json
from pathlib import Path

SESSION_CACHE = Path(".claude/session_cache.json")

def save_session(work_item_id: str, session_id: str) -> None:
    cache = json.loads(SESSION_CACHE.read_text()) if SESSION_CACHE.exists() else {}
    cache[work_item_id] = session_id
    SESSION_CACHE.write_text(json.dumps(cache, indent=2))

def load_session(work_item_id: str) -> str | None:
    if not SESSION_CACHE.exists():
        return None
    return json.loads(SESSION_CACHE.read_text()).get(work_item_id)
```

To resume a WIO session after a CLI restart, pass `resume=load_session(work_item_id)` in `ClaudeAgentOptions`. If `load_session()` returns `None`, the SDK starts a fresh session automatically.

### Context Window Strategy

**WIO (primary concern):**

Claude Sonnet 4.6 has a 200K token context window. The system prompt after Phase 5 skill injection is approximately:
- Skills 02–06 (Phase 3 validated): ~18KB → ~4,500 tokens
- Skills 10+11 (Phase 5 addition, ~7KB per 05-CONTEXT.md): → ~1,750 tokens
- Total system prompt: ~25KB → ~6,250 tokens (3.1% of window)

This is safe. Context pressure only becomes relevant during multi-cycle QA loops where tool results, file reads, and QA findings accumulate across many turns.

**When to act on context pressure:**

Monitor `ResultMessage.usage["input_tokens"]` after each `client.receive_response()` drain:

```python
if isinstance(msg, ResultMessage) and msg.usage:
    input_tokens = msg.usage.get("input_tokens", 0)
    if input_tokens > 120_000:  # 60% of 200K
        # Log warning; consider re-injecting requirements fresh on next cycle
        pass
```

**Three mitigation options (escalating cost):**

Option 1 — Skill index fallback (lowest friction, established in Phase 3): Replace the full skill 10+11 content in the system prompt with brief one-line pointers. The WIO reads the full skill file on demand via the `Read` tool at the start of the relevant lifecycle step.

```python
# In system_prompt assembly — skill index approach when context pressure detected
skill_10_pointer = (
    "Skill 10 (Knowledge Enrichment): Before Builder, "
    "use Read to load '.claude/skills/knowledge-context-enrichment/SKILL.md' and follow it."
)
skill_11_pointer = (
    "Skill 11 (Knowledge Storage): After QA, "
    "use Read to load '.claude/skills/knowledge-storage/SKILL.md' and follow it."
)
```

Option 2 — Stateless re-injection per cycle (prevents compaction loss): At the start of each QA fix cycle, explicitly re-pass the original Linear work item description, acceptance criteria, and implementation notes in the prompt — not as conversational memory but as fresh structured input. This is the primary defence against PITFALLS.md Pitfall 5 (context exhaustion losing requirements specificity).

Option 3 — Fork session for each QA fix cycle: Pass `fork_session=True` in `ClaudeAgentOptions` to create a new session that starts with the accumulated history but diverges from that point. Each fork gets its own session ID. Use when a single work item is consuming more than 3 QA cycles and the conversation history is very long.

**UAT Agent and Risk-14:** Both are bounded single-call sessions with low context footprint. No special management needed.

**Automatic SDK compaction:** The SDK compacts older turns when context approaches capacity. Compaction is lossy for specific requirements stated early in a session. This is why the stateless re-injection pattern is the architectural recommendation for WIO — never rely on conversation history to carry requirements through a multi-cycle lifecycle.

**Extracting and validating SDK output with retry:**

```python
import json
import logging
from pydantic import ValidationError
from claude_agent_sdk import query, ClaudeAgentOptions, ResultMessage
from hsb.contracts.uat import UATResult

logger = logging.getLogger(__name__)
MAX_RETRIES = 3


async def run_uat_and_validate(
    user_story_id: str,
    acceptance_criteria: list[str],
    uat_cycle: int,
) -> UATResult:
    """
    Runs UAT Agent and validates output against UATResult schema.
    Retries up to MAX_RETRIES with the validation error fed back into the prompt.
    Raises RuntimeError after MAX_RETRIES — never returns an unvalidated result.
    """
    skill_08 = load_skill(".claude/skills/uat-validation/SKILL.md")
    schema_json = json.dumps(UATResult.model_json_schema(), indent=2)
    scope_block = "\n".join(f"[AC-{i+1}] {c}" for i, c in enumerate(acceptance_criteria))

    base_prompt = (
        f"Validate User Story {user_story_id} (UAT cycle {uat_cycle}).\n\n"
        "SCOPE BOUNDARY: Only validate the acceptance criteria listed below.\n\n"
        f"Acceptance criteria:\n{scope_block}\n\n"
        f"Return a single JSON object matching this schema:\n{schema_json}\n"
        "Do not include prose before or after the JSON."
    )

    options = ClaudeAgentOptions(
        system_prompt=skill_08,
        allowed_tools=["Read", "Glob", "Grep", "Bash"],
        permission_mode="dontAsk",
        max_turns=20,
        model="claude-sonnet-4-6",
    )

    last_error: str | None = None

    for attempt in range(1, MAX_RETRIES + 1):
        prompt = base_prompt
        if last_error:
            prompt += f"\n\nPrevious attempt failed validation: {last_error}. Fix and retry."

        result_text = ""
        async for msg in query(prompt=prompt, options=options):
            if isinstance(msg, ResultMessage):
                result_text = msg.result or ""

        try:
            # Strip markdown code fences if the model wrapped the JSON
            clean = result_text.strip().removeprefix("```json").removesuffix("```").strip()
            data = json.loads(clean)
            data.setdefault("user_story_id", user_story_id)
            data.setdefault("uat_cycle", uat_cycle)
            return UATResult(**data)
        except (json.JSONDecodeError, ValidationError) as exc:
            last_error = str(exc)
            logger.warning(
                "UATResult validation failed (attempt %d/%d) for %s: %s",
                attempt, MAX_RETRIES, user_story_id, last_error,
            )

    raise RuntimeError(
        f"UAT Agent failed to produce valid UATResult after {MAX_RETRIES} attempts "
        f"for {user_story_id}. Last error: {last_error}"
    )
```

Retry policy summary:
- 3 retries. Each retry feeds the exact validation error text back into the prompt.
- Log every failure at WARNING level with the raw error string.
- After 3 failures, raise — never swallow silently. The Global Orchestrator marks the UAT cycle as `blocked` and flags for human review.

### 4b.2 Async-First Design

The Claude Agent SDK is async-only. There is no synchronous `query()` or synchronous `ClaudeSDKClient`. Every SDK call requires an async context.

**How async works in this SDK:**

```python
# query() — async generator; must be consumed with async for
async for msg in query(prompt="...", options=options):
    if isinstance(msg, ResultMessage):
        result = msg.result

# ClaudeSDKClient — async context manager; client.query() is awaitable
async with ClaudeSDKClient(options=options) as client:
    await client.query("step 1")
    async for msg in client.receive_response():
        ...
    await client.query("step 2")
    async for msg in client.receive_response():
        ...
```

**The one critical mistake — `asyncio.run()` inside a running event loop:**

In Parallel Mode, the Global Orchestrator dispatches multiple WIOs with `asyncio.gather()`. All coroutines run inside the same event loop. Calling `asyncio.run(run_uat_agent(...))` from inside any of those coroutines raises:

```
RuntimeError: This event loop is already running.
```

The fix: every Phase 5 agent entry point is `async def` and called with `await`.

```python
# CORRECT — called from inside an async context (e.g., Global Orchestrator coroutine)
uat_result = await run_uat_and_validate(user_story_id, acceptance_criteria, uat_cycle)

# WRONG — raises RuntimeError when called from inside asyncio.gather()
uat_result = asyncio.run(run_uat_and_validate(user_story_id, acceptance_criteria, uat_cycle))
```

`asyncio.run()` is only legitimate at the top-level CLI entry point (`run_loop.py`). Everywhere else in the application, use `await`.

**Parallel dispatch — existing pattern, unchanged:**

```python
# Global Orchestrator — parallel WIO dispatch (Phase 4 pattern, unchanged in Phase 5)
results = await asyncio.gather(
    *[run_work_item_orchestrator(item.id, item.model_dump_json()) for item in ready_items],
    return_exceptions=True,  # Prevents one WIO failure from cancelling all others
)
```

**Streaming vs. full-result collection:**

Use streaming (`async for msg in query(...)`) as the default for displaying progress. For Pydantic validation, collect the full `ResultMessage.result` before parsing — partial text cannot be parsed as JSON.

```python
# Pattern for streaming display + structured extraction at the end
final_result = ""
async for msg in query(prompt=prompt, options=options):
    if isinstance(msg, AssistantMessage):
        for block in msg.content:
            if isinstance(block, TextBlock):
                print(block.text, end="", flush=True)  # Stream to terminal
    elif isinstance(msg, ResultMessage):
        final_result = msg.result or ""  # Collect full result for Pydantic

uat_data = UATResult(**json.loads(final_result))  # Validate after stream ends
```

### 4b.3 Prompt Engineering Discipline

**System vs. user prompt separation:**

`system_prompt` carries behavioral instructions (skill content, scope constraints, output format rules). The `prompt` argument to `query()` or `client.query()` carries dynamic data (Linear issue JSON, acceptance criteria, QA findings). Never mix rules and data in the same string — it makes skill updates require application code changes.

```python
# CORRECT — rules in system_prompt, data in prompt
options = ClaudeAgentOptions(
    system_prompt=load_skill(".claude/skills/uat-validation/SKILL.md"),
)
await client.query(f"Work item: {work_item.model_dump_json()}")

# WRONG — behavioral instructions embedded in the prompt
await client.query(
    "You are a UAT validator. Your rules are: ... Now validate: " + work_item_json
)
```

**Scope boundary injection (anti-scope-creep, PITFALLS.md Pitfall 8):**

The UAT Agent's prompt must include an explicit, literal scope constraint — not a vague instruction. The constraint must appear in the prompt (not only in the skill file) so it is present regardless of context compaction:

```python
scope_constraint = (
    "SCOPE BOUNDARY: Only validate the acceptance criteria listed below. "
    "Do not evaluate any feature, behavior, or quality dimension not explicitly listed. "
    "Any finding that lacks a direct reference to a listed [AC-N] criterion is out of scope "
    "and must not appear in your response."
)
```

**Few-shot examples — inline only at Phase 5:**

The Knowledge Store is file-based. There is no vector DB for dynamic few-shot retrieval. Add 1–2 concrete examples of correct UAT output format and correct Knowledge Store entry format directly in the respective skill markdown files. Dynamic few-shot retrieval from the Knowledge Store is a Phase 6+ concern.

**Output format instruction — always explicit:**

The model must know it should return JSON, not prose. Include the instruction and the schema in every prompt that requires structured output:

```python
schema_instruction = (
    "\n\nReturn your response as a single JSON object. "
    "Do not include any prose before or after the JSON.\n"
    "Schema:\n" + json.dumps(UATResult.model_json_schema(), indent=2)
)
```

**`max_turns` — never unset in production:**

The SDK default is unlimited turns. An agent hitting a persistent tool error loops until context exhaustion or process kill — no error raised, no warning. Always set `max_turns` explicitly and handle `stop_reason == "error_max_turns"` in the caller.

### 4b.4 Context Window Management

**WIO long-running session (the only active concern in Phase 5):**

Context budget estimate post Phase 5 skill injection:
- System prompt (~6,250 tokens, ~3% of 200K window) — safe, as confirmed in 05-CONTEXT.md
- Per-turn accumulation: file reads (~500–2,000 tokens each), QA findings (~1,000–3,000 tokens), implementation notes (~500–1,500 tokens)
- A 3-cycle QA loop on a large task can consume 80K–120K input tokens — still within 200K but approaching the 60% monitoring threshold

**Context monitoring:**

```python
async for msg in client.receive_response():
    if isinstance(msg, ResultMessage) and msg.usage:
        input_tokens = msg.usage.get("input_tokens", 0)
        cache_hits = msg.usage.get("cache_read_input_tokens", 0)
        if input_tokens > 120_000:
            logger.warning(
                "WIO context at %d tokens (%.0f%% of 200K) for %s",
                input_tokens, input_tokens / 2_000, work_item_id,
            )
```

**Mitigation — stateless re-injection per QA fix cycle:**

Never rely on conversation history to carry original requirements through a multi-cycle QA loop. The SDK's automatic compaction (triggered when context approaches capacity) summarizes older turns and loses specificity. At the start of each QA fix cycle, explicitly re-pass the original acceptance criteria and work item description as fresh structured input:

```python
fix_cycle_prompt = (
    f"Beginning fix cycle {cycle} for {work_item_id}.\n\n"
    "ORIGINAL REQUIREMENTS (re-injected — do not rely on earlier context):\n"
    f"{acceptance_criteria_json}\n\n"
    "QA FINDINGS FROM PREVIOUS CYCLE:\n"
    f"{qa_findings_json}\n\n"
    "Implement fixes for all blocking findings."
)
```

**Skill index fallback if context is tight:**

Replace full skill 10+11 content with one-line pointers when context pressure is detected. The WIO reads the full skill file on demand at the relevant lifecycle step. Trade: one additional `Read` tool call per step vs. ~1,750 tokens of upfront system prompt.

**UAT Agent:** Bounded single call, <30K tokens typical. No active management needed.

**Risk Agent skill 14:** ~3–5K tokens. Negligible.

**Knowledge Store retrieval (Glob+Grep):** At MVP scale (~50 entries), grep output is token-efficient — only matching file sections load into context. If the store grows to ~500 entries, add category-filtering instructions to skill 10 to limit grep scope (e.g., "only search `architecture/` and `qa/` categories for this task type").

**Prompt caching:**

The SDK transparently uses Anthropic prompt caching for stable system prompt content. Skill markdown files injected as `system_prompt` are identical across WIO invocations — they will be cached after the first call, reducing repeated token costs by 80–90% on subsequent calls within the same cache window. ```python
if isinstance(msg, ResultMessage) and msg.usage:
    cache_read = msg.usage.get("cache_read_input_tokens", 0)

## 5. Evaluation Strategy

Phase 5 has **three distinct AI sub-systems** with different eval profiles. The dimensions table below is grouped by sub-system because the rubric language, measurement approach, and acceptable failure rates differ materially across them. Rubrics are translated directly from the domain ingredients in Section 1b — wording is preserved where the source rubric was already operational.

### Dimensions

#### Sub-system A — Intelligence Agent (inline in WIO, skills 10+11)

| # | Dimension | Rubric (Pass / Fail) | Measurement | Priority |
|---|-----------|----------------------|-------------|----------|
| A1 | **Knowledge entry signal threshold** (skill 11 ingestion) | **PASS:** Entry (a) applies to a class of future tasks identifiable by technology, pattern, or domain (not just the specific issue), AND (b) provides a concrete recommendation that changes how a future Builder would approach the work. Cites Linear issue ID and PR URL as evidence. **FAIL:** Entry records a single-occurrence bug fix with no indication of recurrence ("fixed null pointer in method X"); records an observation without a recommendation; or duplicates an existing entry with different wording. | **LLM Judge** (calibrated against developer-operator). Calibration target: ≥ 0.7 Cohen's kappa with operator labels on 20 reference entries before trusting in CI. | **Critical** |
| A2 | **Knowledge entry schema completeness** (INTL-03) | **PASS:** Entry contains all 8 required fields: `title`, `type`, `context`, `evidence` (Linear issue + PR + files), `insight`, `recommendation`, `applicability`, `date`. `applicability` is non-empty AND not the literal string "all tasks". `evidence.linear_issue` matches `LIN-\d+` regex. `evidence.pr` matches `https://github.com/.+/pull/\d+`. **FAIL:** Any required field missing, empty, or matches the forbidden literal "all tasks" / "n/a" / "TBD". | **Code** (pydantic + regex post-write hook on `knowledge/**/*.md`). | **Critical** |
| A3 | **Enrichment retrieval relevance** (skill 10) | **PASS:** Retrieved Knowledge Store entries shown to Builder are judged "applicable to the current work item" by the developer-operator review (sampled). Retrieval precision ≥ 60% on the reference dataset (i.e., of N retrieved entries, ≥ 0.6N are rated relevant). **FAIL:** ≥ 50% of retrieved entries are rated irrelevant by operator (the failure threshold from the Section 1b "Entry reusability" rubric). | **LLM Judge** (cheap pre-screen) + **Human** (operator monthly sampling for ground truth). | **High** |
| A4 | **Knowledge Store deduplication** | **PASS:** New entry's `title` and `insight` text do not have ≥ 0.85 Jaccard overlap with any existing entry in the same category. **FAIL:** Near-duplicate written without superseding the original. | **Code** (Jaccard token-set comparison in pre-write hook; threshold tunable). | **Medium** |

#### Sub-system B — UAT Agent (standalone SDK session, skill 08)

| # | Dimension | Rubric (Pass / Fail) | Measurement | Priority |
|---|-----------|----------------------|-------------|----------|
| B1 | **Acceptance-criteria coverage completeness** (UATA-02) | **PASS:** For every acceptance criterion `[AC-N]` listed in the prompt, the `UATResult.scenarios` array contains at least one scenario with matching `criterion_id` and a non-null `status`. Coverage = 100%. **FAIL:** One or more acceptance criteria appear in the User Story prompt but have no matching scenario in the report; OR `overall_status == "approved"` while coverage < 100%. | **Code** (set-comparison: `{ac.id for ac in prompt} == {s.criterion_id for s in result.scenarios}`). | **Critical** |
| B2 | **Scenario specificity vs. paraphrase mismatch** | **PASS:** Each scenario's `evidence` field is ≥ 10 chars (pydantic min_length) AND describes specific observable behavior (file path, command output, screen state, API response) — not a paraphrase of the criterion. `finding` is non-null whenever `status == "fail"`. **FAIL:** `evidence` paraphrases the criterion ("system supports the X feature as expected"); OR a "fail" scenario has no `finding` text describing the gap. | **LLM Judge** (calibrated against senior QA engineer on the reference dataset). Code-side pre-check: pydantic min_length + regex against paraphrase patterns. | **Critical** |
| B3 | **Scope boundary enforcement** (UATA-04, Pitfall 8) | **PASS:** `UATResult.scope_violations` is empty. No scenario references behaviors, code quality, lint, architecture, or features outside the supplied `[AC-N]` list. No finding text contains banned tokens (`refactor`, `code quality`, `naming`, `style`, `linter`, `inefficient`, `should also handle`, `future edge case`). **FAIL:** Any item in `scope_violations` OR any banned-token match in finding text. | **Code** (banned-token regex on findings; pydantic non-empty check on `scope_violations`) + **LLM Judge** (subtle scope creep where banned tokens are absent but commentary is structural rather than behavioral). | **Critical** |
| B4 | **UAT verdict correctness (false-pass / false-fail)** | **PASS:** UAT `overall_status` matches the human-labeled ground-truth verdict on ≥ 90% of reference dataset User Stories. **False-pass rate ≤ 5%** (worse failure mode — defective work declared done). **False-fail rate ≤ 15%** (creates spurious fix subtasks; consumes operator time but does not erode trust as severely). **FAIL:** Either threshold breached on the rolling reference dataset run. | **LLM Judge** + **Human** (developer-operator + senior QA engineer label the 20 reference User Stories with ground-truth verdicts). | **Critical** |
| B5 | **UAT cycle accounting** (Pitfall 2 extension to UAT) | **PASS:** `uat_cycle` field increments by exactly 1 each retrigger; total UAT cycles per User Story ≤ 3 (project-wide `max_qa_cycles` invariant applied to UAT). **FAIL:** UAT triggered on a User Story already at cycle 3 without operator override. | **Code** (Global Orchestrator pre-dispatch check; Linear field assertion). | **High** |
| B6 | **Output schema validity** | **PASS:** `ResultMessage.result` parses as JSON and validates against `UATResult` pydantic model on first attempt. **FAIL:** Validation failures requiring retry (per Section 4b.1 retry policy: 3 attempts max, then RuntimeError). Track first-attempt validity rate ≥ 95%. | **Code** (pydantic `ValidationError` count via SDK retry wrapper). | **High** |

#### Sub-system C — Risk Agent (pure Python skills 12+13 + LLM skill 14)

| # | Dimension | Rubric (Pass / Fail) | Measurement | Priority |
|---|-----------|----------------------|-------------|----------|
| C1 | **Quality score determinism / reproducibility** (skill 12) | **PASS:** Running `risk_agent.calculate_quality_score(work_item)` twice on identical input returns identical `QualityScore` (byte-equal `score_breakdown` dict). The score formula (start=100, -10 per QA failure, -5 per fix subtask, -15 if UAT failed, -5 per rework cycle, min=0) is the only source of the value. **FAIL:** Two consecutive runs differ for the same input; or score changes when underlying QA/UAT history has not changed. | **Code** (property-based test with Hypothesis; idempotency assertion; pure-function unit tests). | **Critical** |
| C2 | **Score traceability (operator can audit)** | **PASS:** Every `QualityScore` includes a populated `score_breakdown` dict showing per-component penalties (e.g., `{"qa_failures": -20, "fix_subtasks": -10, "uat_failed": 0, "rework": -5}`) with a sum that equals `100 - score`. **FAIL:** `score_breakdown` is empty, missing components, or its values do not arithmetically reconcile to the displayed `score`. | **Code** (pydantic + arithmetic assertion in skill 12 unit tests: `100 - sum(breakdown.values()) == score`). | **High** |
| C3 | **Priority queue tiebreaker stability** | **PASS:** When two work items have identical scores, ordering is determined by a documented tiebreaker (Linear `updatedAt` ascending — older first). The same input list always produces the same output ordering. **FAIL:** Same-score items appear in nondeterministic order across runs; no documented tiebreaker. | **Code** (deterministic-sort unit test with two equal-score items). | **High** |
| C4 | **Priority queue face validity (operator-judged)** | **PASS:** On 10–15 backlog items labeled by an engineering team lead with ground-truth priority order, the Risk Agent's queue achieves Spearman rank correlation ≥ 0.7. **FAIL:** Correlation < 0.5 (operator would reroute manually rather than follow the queue — the "score gaming / dismissal" failure mode in Section 1b). | **Human** (team lead labels reference backlog quarterly; Spearman computed in CI eval suite). | **High** |
| C5 | **Auto-Improvement Trigger precision** (skill 14, RISK-03) | **PASS:** Operator approves ≥ 60% of suggested triggers (i.e., agrees the pattern is real and worth a Linear work item). Each `AutoImprovementTrigger.pattern_evidence` lists ≥ 2 distinct work item IDs or QA finding refs. **FAIL:** Operator dismissal rate > 60% over a 4-week rolling window; OR `pattern_evidence` has < 2 supporting refs. | **LLM Judge** (cheap pre-screen for evidence completeness) + **Human** (operator approval/dismissal logged in Linear via comment, sampled monthly). | **High** |
| C6 | **Auto-Improvement Trigger output never carries `linear_state` other than `"suggested"`** | **PASS:** `AutoImprovementTrigger.linear_state == "suggested"` for every emitted trigger (pydantic `Literal["suggested"]` enforces this). **FAIL:** Any value other than `"suggested"` produced (would be a contract violation; pydantic should reject it before emission). | **Code** (pydantic Literal type — compile-time + runtime guarantee). | **Critical** |

> **Cross-cutting dimensions** (apply to all three sub-systems): output schema validity (already encoded as B6, A2, C2, C6 above) and tool-use safety (no `Agent` tool ever appears in `allowed_tools`; no MCP appears for Risk-14 — both verified at session-config layer in Section 6 online guardrails).

### Reference Dataset

**Size:** 30–60 examples total across the three sub-systems, broken down as:

| Sub-system | Count | Composition |
|------------|-------|-------------|
| Intelligence (A) | 10–20 | 5 critical-path retrievals (relevant entry exists and should be retrieved); 5 negative-path (no relevant entry — agent should report "no enrichment available"); 5 ingestion-decision examples (real QA findings: 3 worth persisting, 2 noise); 5 near-duplicate detection cases. |
| UAT (B) | 10–20 | 5 critical-path (clean User Stories that should pass UAT); 5 explicit-fail (1+ acceptance criterion provably unmet); 3 ambiguous-criteria edge cases (acceptance criteria with subjective wording — labelers must agree on ground truth); 3 scope-creep adversarial (PR contains tempting code-quality issues that UAT must not flag); 2 acceptance-criteria gap (criterion silently absent from scenarios — false-pass test). |

**Composition principles** (per ai-evals.md "start small, expand from real failure modes"):
- **Real Linear data first.** Harvest reference examples from the operator's actual prior User Stories and PRs wherever possible — this is the project's "real data, no mocking" stance (`02-CONTEXT.md` integration test strategy) applied to evals. Do not invent synthetic User Stories when historical ones exist.
- **Failure modes are first-class.** Every Section 1b failure mode must have at least one positive (passes the rubric correctly) and one negative (fails as expected) example: false-pass UAT, scope-bleed UAT, knowledge graveyard, opaque risk score.
- **Adversarial inputs.** PRs that contain code smells UAT must ignore (B3); Knowledge Store with stale entries (A3 retrieval should still succeed); backlogs with all-zero scores (C3 tiebreaker test).

**Construction timeline:** Build the reference dataset **concurrently with implementation**, not after. The first 5 examples per sub-system are written before the corresponding agent code is written — they serve as the agent's acceptance criteria.

**Labeling:**

| Dataset Slice | Primary Labeler | Calibrator |
|--------------|----------------|------------|
| A1 (Knowledge entry signal) | Developer-operator | Self-calibration: re-label same 5 entries 1 week apart; require ≥ 0.8 self-agreement before trusting LLM judge against this label set. |
| A3 (enrichment relevance) | Developer-operator (production sampling) | Bootstrap label set built during implementation; updated monthly from production samples. |
| B1, B2, B6 (coverage / specificity / schema) | Code (deterministic) | n/a — no human label required. |
| B3 (scope creep) | Code (regex) + senior QA engineer for subtle cases | QA engineer labels the 3 ambiguous-criteria edge cases. |
| B4 (UAT verdict correctness) | Developer-operator + senior QA engineer (joint label session) | Disagreements between operator and QA engineer are resolved by adding a third "ambiguous" label and excluding from accuracy computation. |
| C1, C2, C3, C6 (deterministic Risk) | Code (pytest + Hypothesis) | n/a. |
| C4 (priority queue face validity) | Engineering team lead / technical PM | Ground-truth re-labeled quarterly; correlation drift between quarters is itself a signal. |
| C5 (Auto-Improvement Trigger precision) | Developer-operator (in production via Linear comment approve/dismiss) | Trailing 4-week rolling window; LLM judge spot-checks evidence completeness pre-emission. |

**LLM judge calibration protocol** (applies to A1, B2, B4, C5):
1. Domain expert labels first 10 reference examples by hand.
2. LLM judge runs against the same 10 examples.
3. Compute Cohen's kappa (binary rubrics) or Spearman ρ (ordinal).
4. Trust threshold: ≥ 0.7. Below 0.7 → revise the judge prompt; never trust an uncalibrated judge in CI.
5. Re-calibrate quarterly or after any judge prompt change.

## 6. Guardrails

The classification rule (per ai-evals.md "Guardrail vs. Flywheel Decision"): if the failure is **catastrophic** (silent Linear pollution, unauthorized writes, runaway loops, broken authentication) → online guardrail. If the failure is a **quality signal** that improves over time with sampling → offline flywheel.

### Online (Real-Time)

These run on every relevant invocation. Each is intentional — every guardrail adds latency and complexity. The list is deliberately short.

| # | Guardrail | Trigger Point | Detection | Intervention | Source |
|---|-----------|---------------|-----------|--------------|--------|
| G1 | **No `ANTHROPIC_API_KEY` present (OAuth-only)** | Process startup (`hsb/cli.py`, `run_loop.py`, every CI eval entry point) | `assert "ANTHROPIC_API_KEY" not in os.environ` | **Block** — raise SystemExit with explicit error message; refuse to start. | Pitfall 6 in Section 3; project-wide rule (memory). |
| G2 | **No `Agent` tool in `allowed_tools` (WORC-02 structural)** | Every `ClaudeAgentOptions` construction site | `"Agent" not in options.allowed_tools` validated by helper factory `make_options()` in `src/hsb/agents/_sdk_options.py` | **Block** — raise `ValueError` before SDK call; cannot be silently bypassed. | WORC-02; failure mode #4 in Section 1. |
| G3 | **No Task-tool dispatch in session log (runtime backstop for G2)** | Hook on every `ResultMessage` | Inspect `usage` and message log for any `tool_use_block` with `name == "Task"` | **Block + alert** — abort the WIO session; emit `WARN` log; flag for operator review. | Defense-in-depth: G2 catches misconfiguration; G3 catches an SDK regression that bypasses `allowed_tools`. |
| G4 | **Risk Agent skill 14 has zero tools and zero MCP** | `detect_auto_improvement_triggers()` SDK call site | `assert options.allowed_tools == []` and `assert options.mcp_servers is None` | **Block** — raise `RuntimeError("RISK-04: skill 14 misconfigured")` before SDK call. | RISK-04; failure mode #2. | G5 | **No Linear write call from Risk Agent code path** | All `linear_agent.write_*` methods | Inspect Python call stack: if any frame originates from `src/hsb/agents/risk_agent.py`, **except** the explicit operator-delegated path through `src/hsb/agents/global_orchestrator.py::approve_improvement_trigger()`, deny | **Block + alert** — raise `PermissionError("RISK-04 violation")`; log full stack trace. | RISK-04; failure mode #2. Implemented as a decorator on `LinearAgent` write methods. |
| G6 | **UAT cycle cap (max 3, project-wide invariant)** | Global Orchestrator UAT-readiness check, before re-dispatching UAT after fix subtasks complete | `linear.get_issue(user_story_id).uat_cycle < 3` | **Block + escalate** — do not re-dispatch UAT; create a Linear comment "UAT cycle cap reached; awaiting operator decision"; flag for human review. | Pitfall 2 in `PITFALLS.md` (extended from QA to UAT); failure mode #6. |
| G7 | **WIO `max_turns` and `stop_reason` checked** | After every `ResultMessage` in WIO, UAT, Risk-14 | `msg.stop_reason != "error_max_turns"` | **Flag + escalate** — do not silently continue; raise `RuntimeError`; mark the work item / User Story as `blocked` for human review. | Pitfall 5 in Section 3; loop detection. |

Does not block. | Pitfall 5 in `PITFALLS.md` (context exhaustion); failure mode #5. |
| G9 | **Knowledge entry passes pre-write hook** | Before any `Write` to `knowledge/**/*.md` from skill 11 | Pydantic validation of `KnowledgeEntry` model + regex check on `evidence.linear_issue` (`LIN-\d+`) and `evidence.pr` (`https://github.com/.+/pull/\d+`) + Jaccard similarity check (A4) | **Block** — refuse the write; return validation error to the WIO session so the LLM can revise or skip. | INTL-02, INTL-03; failure mode #3. |
| G10 | **UAT output passes schema + scope check before persistence** | Before `LinearAgent.update_uat_status()` is called | Pydantic `UATResult` validation + B1 coverage check (set-equality on `criterion_id`) + B3 banned-token regex on findings | **Block** — refuse the persist call; surface a structured error to the operator. | UATA-02, UATA-04; failure mode #1. |

### Offline (Flywheel)

These do not block any individual call. They feed system improvement over time via sampled review and trend detection.

| Metric | Sampling Strategy | Cadence | Action on Degradation |
|--------|-------------------|---------|------------------------|
| **UAT verdict false-pass rate** (lagging indicator) | Every UAT verdict where the operator subsequently re-opens the User Story or flips status from `done` → `in_progress` within 14 days is auto-flagged. Plus 10% random sampling of approved verdicts. | Weekly review | If false-pass rate > 5% on rolling 4-week window → revise skill 08 prompt (specifically the scope-boundary and observable-evidence sections); re-calibrate B4 LLM judge. |
| **UAT false-fail rate** | Every UAT-driven fix subtask the operator closes as "won't fix" or "invalid" within 7 days of creation is auto-flagged. | Weekly review | If false-fail rate > 15% → tighten scope-creep detection; review the 5 most recent false-fails for common patterns. |
| **Knowledge Store enrichment hit rate** | Every Builder run logs which Knowledge Store entries (if any) were cited in the implementation notes. Computed: `entries_cited / entries_retrieved`. Operator-flagged irrelevant entries (false-positives) are tagged in the entry's frontmatter. | Bi-weekly review | If hit rate < 30% on a 2-week window → review skill 10 retrieval prompt; consider adding category filtering; reconsider deferred ADVL-01 (semantic search). |
| **Knowledge Store entry early-deletion rate** | Any entry deleted by the operator within 7 days of write is auto-flagged as a "graveyard signal" (Section 1b failure mode #2). | Weekly review | If early-deletion rate > 20% → tighten skill 11 ingestion criteria (signal threshold rubric A1); re-calibrate the LLM judge. |
| **Risk Agent priority queue override rate** | Every cycle where the operator picks a work item that is **not** in the top-3 of the Risk Agent's queue is logged. | Monthly review | If override rate > 50% → re-label C4 reference dataset with team lead; re-tune skill 12 penalty weights. |
| **Auto-Improvement Trigger acceptance rate** (C5 production metric) | Every emitted trigger is logged in Linear as a comment with approve/dismiss reactions; trailing 4-week rolling window. | Monthly review | If acceptance rate < 60% → revise skill 14 prompt (tighten pattern-evidence requirement to ≥ 3 refs); reduce trigger emission rate. |
| **WIO context budget approach rate** | Cycles where `input_tokens > 120K` (G8 warning fired) are aggregated. | Weekly review | If > 10% of WIO cycles trigger the warning → activate the skill-index fallback (Section 4.b.4) by default; consider Option 2 stateless re-injection always-on. |
| **Prompt cache hit rate** | `usage.cache_read_input_tokens / usage.input_tokens` per WIO call. | Weekly review | If cache hit rate < 70% on the system prompt portion → audit `system_prompt` for dynamic content leak (Section 4.b.5); fix and re-deploy. |
| **First-attempt schema validity rate** | `UATResult` and `AutoImprovementTrigger` retry counter: `validations_passed_first_attempt / total_validations`. | Weekly review | If < 95% → revise the schema instruction in the prompt (Section 4.b.3); add a stronger few-shot example. |

## 7. Production Monitoring

**Tracing tool:** **Arize Phoenix**, self-hosted at `http://phoenix.internal:6006` (or local at `http://localhost:6006` for solo developer-operator mode). Spans emitted by the Claude Agent SDK via OTel are auto-ingested. No third-party platform credential — satisfies OAuth-only constraint.

**What gets traced** (every span auto-tagged with `work_item_id`, `agent` ∈ {`wio`, `uat`, `risk_14`}, `session_id`, `model`):

- Every `query()` and `client.query()` call: input prompt, system prompt token count, output text, `usage` dict (input/output/cache tokens), `total_cost_usd`, `stop_reason`, `num_turns`, latency.
- Every Pydantic validation outcome (pass / fail with error message).
- Every Linear MCP tool call: tool name, arguments, response status.
- Every Knowledge Store read (Glob/Grep) and write event with file path.
- Every guardrail fire (G1–G10): which guardrail, work_item_id, intervention taken.

### Key Metrics to Track

The 5 metrics that drive operator decisions (per ai-evals.md "only track metrics that drive decisions"):

| # | Metric | Source | Why It Matters |
|---|--------|--------|----------------|
| M1 | **Unauthorized Linear writes from Risk Agent** | G5 guardrail counter | RISK-04 absolute invariant. Must be **0**. Any non-zero value is a critical incident. |
| M2 | **UAT false-pass rate (rolling 14-day)** | Operator override of approved User Stories (status flip `done` → `in_progress` or re-open within 14 days) | Failure mode #1. The trust-collapse vector for the entire UAT Agent. |
| M3 | **WIO `error_max_turns` rate** | `ResultMessage.stop_reason` aggregation | Pitfall 5 / loop detection. Indicates either tool failures or context exhaustion — both require investigation. |
| M4 | **Knowledge Store contamination signal** | Composite of: entries deleted within 7 days + operator-flagged irrelevant retrievals + entries failing the A2 schema check post-hoc | Failure mode #3. Lagging indicator of skill 11 quality drift. |
| M5 | **Cost per work item** | Phoenix span aggregation: sum `total_cost_usd` per `work_item_id` | Cost drift detection. Baseline ~$0.20–$0.40 per WIO with caching (Section 4.b.5). Sustained > $1.00 indicates context-budget regression or cache miss. |

### Alert Thresholds

Alerts page the developer-operator (Slack/email — operator chooses). Thresholds tuned for the small-team / personal-tooling scale; a regulated-industry deployment would tighten them.

| Alert | Threshold | Severity | Action |
|-------|-----------|----------|--------|
| **A-CRIT-1** | M1 (unauthorized Linear write) > 0 in any 24h window | **Critical** | Page immediately; halt Risk Agent until incident review. |
| **A-CRIT-2** | G1 (`ANTHROPIC_API_KEY` present) fires at process startup | **Critical** | Process refuses to start; alert is in the SystemExit message. |
| **A-CRIT-3** | M2 (UAT false-pass rate) > 5% on rolling 14-day window | **Critical** | Page; UAT Agent enters "advisory only" mode (verdicts logged but do not auto-close User Stories) until skill 08 revision. |
| **A-WARN-1** | M3 (`error_max_turns` rate) > 5% of WIO calls in rolling 7-day window | Warning | Slack notification; investigate top 3 work items by turn count. |
| **A-WARN-2** | M4 (Knowledge contamination signal) > 20% in rolling 14-day window | Warning | Slack notification; trigger Knowledge Store audit (operator reviews flagged entries). |
| **A-WARN-3** | M5 (cost per work item) > $1.00 average over rolling 7-day window | Warning | Slack notification; investigate prompt cache miss rate (G8 + cache flywheel metric). |
| **A-INFO-1** | Auto-Improvement Trigger dismissal rate > 60% (C5 flywheel) | Info | Weekly digest; revise skill 14 prompt next iteration. |
| **A-INFO-2** | Risk priority queue override rate > 50% (Risk flywheel) | Info | Weekly digest; re-label C4 reference dataset. |

### Smart Sampling Strategy

Per ai-evals.md "smart sampling — weight toward interactions with concerning signals." Production traffic is too low to label everything, and uniform random sampling wastes operator review time on uneventful cases. The strategy weights operator-attention-worthy interactions for next-day review:

| Signal | Sample Rate | Rationale |
|--------|-------------|-----------|
| **Operator override** of any agent verdict (UAT pass rejected, Risk priority dismissed, Knowledge entry deleted < 7 days post-write) | 100% — auto-flagged | The strongest possible quality signal — the operator has already done the human label work. Each override becomes a candidate for the reference dataset. |
| **Guardrail fire** (G1–G10) | 100% — auto-flagged with full trace | Catastrophic-failure precursors; investigate every instance. |
| **WIO `error_max_turns`** | 100% — auto-flagged | Loop / tool-failure indicator; full trace review required. |
| **WIO context > 120K tokens** (G8) | 100% — flagged for review | Pitfall 5 precursor; review whether stateless re-injection (Section 4.b.4 Option 2) should activate for this work item type. |
| **UAT verdict on a User Story with > 5 acceptance criteria** | 100% — flagged | High-coverage UATs are highest-stakes; manual review confirms B1 coverage in practice, not just on paper. |
| **Risk Agent score = 0 or 100 boundary** | 100% — flagged | Boundary conditions surface formula bugs; cheap to review. |
| **Successful WIO completions with all green** | 5% — random sample | Confirms baseline behavior; prevents "we only review failures" bias. |
| **All other interactions** | 1% — random sample | Long-tail quality drift detection. |

**Review queue:** Phoenix's "Annotations" feature is used as the review surface — flagged spans appear in a daily review queue ordered by signal strength. The operator's review labels (correct / incorrect / ambiguous) feed back into the reference dataset and the LLM-judge calibration protocol (Section 5).

**Signal-metric divergence detection:** A monthly review compares the metrics in this section against operator behavior signals (override rate, manual reordering frequency, time-to-act-on-recommendation). Divergence — e.g., metrics look healthy but operator overrides are climbing — is the early-warning system for evaluation gaps. When divergence appears, the gap drives the next reference dataset expansion, not hypothetical coverage.
