<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Linear issue content → @tool wrapper args | Untrusted content authored by humans flows in via `args["issue_content"]` JSON string; mitigated by `json.loads` + Pydantic `BuilderInput` validation, never embedded directly into a prompt string |
| ClaudeAgentOptions config → SDK runtime | Misconfigured options can spawn sub-agents (Pitfall 1); mitigated by enforcing only `mcp_servers` registration and source-grep tests |
| qa_cycle_count from Linear → orchestrator decision | Stale or incorrect cycle count could allow a 4th QA cycle (E-of-P); mitigated by Layer 2 `_check_qa_cycle_cap` safety net post-session |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-01 | Tampering | @tool wrapper args (issue_content as JSON string) | mitigate | `run_builder_tool` parses `args["issue_content"]` via `json.loads`, then constructs Pydantic `BuilderInput` (which has `extra="forbid"` from Phase 2). Any injection attempt that does not match the schema is rejected at the Pydantic boundary BEFORE reaching builder_agent.run_builder_agent. Linear issue content is NEVER concatenated into the system_prompt as raw text. |
| T-3-02 | Information Disclosure | Logging in run_orchestration_cycle and _check_qa_cycle_cap | mitigate | `logger.info` calls only log cost, work_item_id, qa_cycle_count, qa_status. NO logging of `ANTHROPIC_API_KEY`, `LINEAR_API_KEY`, `BuilderInput.repository_context.root_path` (which could leak filesystem paths), or full system_prompt content. |
| T-3-03 | Denial of Service | ClaudeAgentOptions.max_turns | mitigate | `max_turns=30` is hard-coded as a guard against runaway tool calls. Source-grep test asserts the value is present. Adjustable post-MVP if benchmark shows need. |
| T-3-04 | Elevation of Privilege | _check_qa_cycle_cap (Layer 2 of D-05 dual-layer enforcement) | mitigate | When `qa_cycle_count >= 3 and qa_status == "changes_required"`, the function posts a Linear escalation comment and returns. The function itself does not initiate a 4th cycle. Combined with Phase 2 QAOutput model_validator (Layer 1) that rejects `qa_cycle_count >= 3 + changes_required` at the contract boundary, no code path can spawn a 4th QA cycle. |
</threat_model>
