## 1. System Classification

**System Type:** Multi-Agent | Code Automation

**Description:**
Phase 2 builds four independently functional execution agents — Backlog, Builder, Git, and QA — each implemented as a Claude Code SKILL.md skill within `.claude/skills/`. Each agent has strict, non-overlapping tool boundaries enforced by the AgentDefinition scoped-tools list. The system runs on Anthropic Claude (locked in) and integrates natively with the Linear MCP server and `gh` CLI. "Good" looks like: each agent reads its input contract, uses only its allowed tools, produces a correct output contract, and fails loudly rather than silently when constraints are violated.

**Critical Failure Modes:**
1. **Capability boundary bleed** — Builder Agent writes to git or Linear (BLDR-04 violation), QA Agent modifies code (QAAG-05 violation), Git Agent merges to main directly. These destroy the trust model of the entire system.
2. **QA runaway** — QA Agent continues requesting fixes after `qa_cycle_count >= 3` instead of approving with tech-debt annotation. Causes infinite fix loops.
3. **Double-claim in Linear** — Backlog Agent creates duplicate EPICs/tasks in Linear if re-run on the same plan.md without idempotency checks. Pollutes the Linear workspace with phantom work items.
4. **Silent contract failure** — Agent produces output that looks valid but violates the Pydantic schema (e.g., wrong field names, missing required fields). Downstream agents receive malformed input and fail in unpredictable ways.
5. **Stale state on PR rebase** — Git Agent fails to trigger REBASE_STACK for all sibling task PRs when a sibling merges, leaving branches with stale base commits that cause merge conflicts at EPIC PR time.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer tooling / Engineering automation

**User Population:** Solo developer or small engineering team using Claude Code as their primary development environment. Operators invoke agents from the terminal to automate project management and code delivery workflows. Technical users who will read output contracts directly and debug agent behavior via git history and Linear audit logs.

**Stakes Level:** Medium

**Output Consequence:** Incorrect agent output propagates into Linear (corrupted work item hierarchy, phantom tasks, incorrect status), GitHub (orphaned branches, mistargetted PRs, stale rebases), and the codebase (incorrect implementation scope). Errors are recoverable but require manual remediation that undermines the time-saving purpose of the system.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|----------------------|-------------------|--------|--------|
| Linear hierarchy correctness | EPIC → User Story → Task → Subtask created with correct parent-child relationships and traceability quotes from plan.md | Flat task list, missing parent links, wrong hierarchy levels, EPICs without User Stories | High — corrupts Linear as system of record | BKPK-01 to BKPK-05 |

| Fix subtask count | QA creates 0–5 fix subtasks per report | QA creates 6+ fix subtasks in a single report | Medium — exceeds cap defined in requirements | QAAG-03 |

### Known Failure Modes in This Domain

- **Double-claim (Pitfall 1)**: Backlog Agent re-run on same plan.md creates duplicate Linear hierarchy. Must check for existing EPICs with matching title before creating.
- **QA runaway (Pitfall 2)**: QA cycle counter not incremented, or counter not read before deciding whether to request more fixes. Causes unlimited fix cycles.
- **Stale state handoff (Pitfall 4)**: Builder Agent reads cached work item state that was updated by a concurrent QA review. Must fetch fresh state at task start.
- **Silent schema drift**: Pydantic model in `src/hsb/contracts/` diverges from schema defined in `agents/AGENT-CONTRACTS.md`. One is updated without the other. Downstream agents receive unexpected field names.
- **REBASE_STACK partial execution**: Git Agent enumerates sibling PRs via `gh pr list` but misses PRs created after the EPIC branch was established. Must filter by both base branch AND open state.

### Regulatory / Compliance Context

None identified. This is an internal developer tooling system with no PII handling, no regulated data, and no compliance requirements. The only enforcement constraints are the project-defined capability boundaries in REQUIREMENTS.md.

| Role | Responsibility |
|------|---------------|
| Developer / operator | Integration test authorship, contract schema labeling, Linear workspace inspection, acceptance of output contracts |
| Linear workspace admin | Verification that created Linear hierarchy is correct and traceable; cleanup of failed runs |

## 2. Framework Decision

**Selected Framework:** Claude Agent SDK (Claude Code SKILL.md architecture)

**Version:** Claude Code CLI current release; Anthropic SDK `anthropic>=0.40.0`; claude-agent-sdk `0.1.73+`

**Rationale:**
The project is Anthropic/Claude-locked by explicit architectural decision recorded in STACK.md and enforced by REQUIREMENTS.md (LangChain, LangGraph, CrewAI, AutoGen all banned with rationale). The Claude Agent SDK's AgentDefinition scoped-tools list maps directly to the strict per-agent capability boundaries required by BLDR-04, QAAG-05, and GITA-05. Native MCP server configuration handles the Linear MCP integration without additional adapters. SKILL.md files in `.claude/skills/` are the project's established agent deployment pattern (Phase 1 established this with the Linear Agent). The SDK's lifecycle hooks and context compaction handle long tool-call sequences (Backlog Agent creating 20+ Linear issues) without manual context management.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangGraph | Explicitly banned in STACK.md and REQUIREMENTS.md Out of Scope. Overkill for SKILL.md-based agents; graph abstraction overhead without benefit when Claude Code handles the execution loop. |
| CrewAI | Explicitly banned. Python-only but crew abstraction doesn't map to the capability-boundary model where each agent has strict tool restrictions. |
| LangChain | Explicitly banned. Abstraction overhead and complexity creep risk noted in PITFALLS.md. |
| Direct Anthropic Client SDK | Valid fallback if claude-agent-sdk autonomous loop is too opaque for debugging integration tests. Can implement tool loop manually with full call visibility. Add only if SDK debugging proves insufficient. |

**Vendor Lock-In Accepted:** Yes — Anthropic/Claude locked in by architectural decision. The project is Claude Code-native by design; model flexibility is explicitly out of scope.

### Entry Point Pattern
```python
# Claude Code SKILL.md pattern — each agent is a skill file
# .claude/skills/backlog-planning/SKILL.md defines the agent behavior
# Invoked by Claude Code's skill runner when the user calls the skill

# For Python integration layer (src/hsb/agents/backlog_agent.py):
import anthropic
from src.hsb.contracts.backlog import BacklogInput, BacklogOutput

def run_backlog_agent(input: BacklogInput) -> BacklogOutput:
    client = anthropic.Anthropic()
    # Tool definitions passed to Claude for Linear MCP calls
    response = client.messages.create(
        model="claude-opus-4-7",
        max_tokens=8192,
        tools=[...],  # scoped tool list for Backlog agent only
        messages=[{"role": "user", "content": input.model_dump_json()}]
    )
    return BacklogOutput.model_validate_json(extract_final_output(response))
```

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| SKILL.md | Markdown file defining agent behavior, allowed tools, and system prompt — the Claude Code skill format | Defining each of the four agents (Backlog, Builder, Git, QA) |
| AgentDefinition scoped tools | List of MCP tools the agent is permitted to call — enforced by Claude Code skill runner | Enforcing BLDR-04, QAAG-05, GITA-05 capability boundaries |
| Pydantic contracts | Input/output schema for each agent — defined in `src/hsb/contracts/` and validated against `agents/AGENT-CONTRACTS.md` | All agent invocations; ensures downstream agents receive correctly typed input |
| MCP server config | `.claude/settings.json` MCPServers block — configures which MCP servers are available to skills | Linear MCP (npx mcp-remote), filesystem, bash |
| Linear Agent service | `src/hsb/agents/linear_agent.py` — Python wrapper around Linear MCP for QA Agent to write qa_cycle_count and fix subtasks | QA Agent's Linear writes (not direct MCP calls from QA skill) |

### Common Pitfalls
1. **Skill tool scope bleed**: If the SKILL.md `allowed-tools` frontmatter is not explicitly set, Claude Code may offer the full tool set to the agent. Always enumerate exactly what each agent is allowed to call.
2. **Pydantic v1/v2 mismatch**: The project uses Pydantic v2. Do not use `class Config:` or `.dict()` — use `model_config = ConfigDict(...)` and `.model_dump()` instead.
3. **MCP tool name collisions**: Linear MCP and filesystem MCP may expose identically named operations (e.g., `read_file` vs Linear `get_issue`). Always namespace tool calls in SKILL.md to the correct MCP server.
4. **Context compaction mid-task**: Long Backlog Agent runs (20+ Linear issues) may trigger context compaction. Ensure the output contract is written to a file mid-run, not held in context, so compaction does not lose partial results.

## 4. Implementation Guidance

**Model Configuration:**
- Model: `claude-opus-4-7` for Backlog and QA agents (complex reasoning); `claude-sonnet-4-6` acceptable for Git agent (deterministic `gh` CLI commands)
- max_tokens: 8192 for Backlog Agent (generates many Linear issues), 4096 for others
- temperature: 0 (deterministic output contracts — no creative variation wanted)
- No streaming needed; all agents produce a final output contract

**Core Pattern:**
Each agent follows the same three-phase pattern: (1) Read and validate input contract, (2) Execute tool-call loop within allowed tool scope, (3) Produce and validate output contract. The SKILL.md file defines the system prompt and allowed tools; the Python integration layer (`src/hsb/agents/`) handles contract marshaling and validation.

**Tool Use:**
- Backlog Agent: Linear MCP tools (`create_issue`, `create_project`) + Read (plan.md)
- Builder Agent: Read, Edit, Write, Bash (for running validations) — NO git, NO Linear MCP
- Git Agent: Bash (gh CLI commands only) — NO code edits, NO Linear MCP
- QA Agent: Read (PR diff via gh), Linear Agent service (Python import) — NO code edits, NO git

**State Management:**
No persistent state between agent invocations. Each agent reads its input contract at invocation time. `qa_cycle_count` is provided in the input contract by the caller (caller fetches from Linear before invoking QA Agent). Output contracts are written to files in `/tmp/hsb-output/` or passed directly to the CLI caller.

**Context Window Strategy:**
Backlog Agent may process large plan.md files and create 20+ Linear issues. Mitigate by: (1) writing partial output (created issue IDs) to a JSON file after each Linear create call, (2) reading that file at the end to assemble the final output contract rather than holding all results in context. Builder Agent receives a single work item scope — context is bounded.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### Structured Outputs with Pydantic

All agent inputs and outputs are Pydantic v2 models. Schemas are defined in `src/hsb/contracts/` and MUST match `agents/AGENT-CONTRACTS.md` exactly.

```python
# Example: QA Agent output contract (src/hsb/contracts/qa.py)
from pydantic import BaseModel, Field, model_validator
from typing import Literal, Optional

class QAFinding(BaseModel):
    severity: Literal["critical", "high", "medium", "low"]
    blocking: bool
    description: str
    evidence: str  # specific line/diff reference
    suggested_fix_subtask: Optional[str] = None  # Linear subtask title if blocking

class QAOutput(BaseModel):
    qa_status: Literal["approved", "changes_required"]
    qa_cycle_count: int = Field(ge=1, le=3)
    findings: list[QAFinding] = Field(max_length=5)  # hard cap: max 5 findings
    tech_debt_annotation: Optional[str] = None  # present when qa_cycle_count == 3

    @model_validator(mode="after")
    def validate_cycle_cap_logic(self) -> "QAOutput":
        if self.qa_cycle_count >= 3 and self.qa_status == "changes_required":
            raise ValueError("At qa_cycle_count >= 3, status must be 'approved' with tech_debt_annotation")
        if self.qa_cycle_count >= 3 and not self.tech_debt_annotation:
            raise ValueError("tech_debt_annotation required when qa_cycle_count >= 3")
        return self
```

On validation failure, the integration layer raises and the CLI outputs a structured error — never silently passes a malformed contract downstream.

### Async-First Design

The Anthropic SDK supports async (`AsyncAnthropic`), but Phase 2 agents are invoked as CLI commands (synchronous). Use synchronous client for all Phase 2 agents. Async is appropriate only if agents are later exposed as FastAPI endpoints (Phase 4+). The one mistake: do not mix `asyncio.run()` inside a synchronous call stack — choose one model and stick to it.

### Prompt Engineering Discipline

- **System prompt** (SKILL.md header): agent identity, capability boundaries, output contract format, hard constraints (capability boundary rules, cycle cap). This is immutable per agent.
- **User prompt**: the serialized input contract (`input.model_dump_json()`). Never concatenate free-form text here.
- **Few-shot examples**: include one concrete example of correct EPIC → User Story → Task → Subtask structure in the Backlog Agent system prompt. Reduces hierarchy errors.
- **Token budget**: system prompt < 2000 tokens, input contract < 4000 tokens. If plan.md exceeds 4000 tokens, chunk by section and run Backlog Agent per section.

### Context Window Management

For Backlog Agent handling large plan.md files:
1. Split plan.md by top-level heading (each `#` section is one chunk)
2. Run Backlog Agent on each chunk → produces partial output contract
3. Merge partial contracts into final BacklogOutput by deduplicating EPIC titles
4. Write final contract to file; do not hold all intermediate results in context

For QA Agent reviewing large PR diffs:
1. Limit diff to changed files relevant to the Linear work item scope
2. Use `gh pr diff --name-only` to identify files, then `gh pr diff -- <file>` per file
3. Never pass full monorepo diff to QA Agent

### Cost and Latency Budget

- Backlog Agent: ~20-40 tool calls per plan.md (one per Linear issue created) → ~$0.10-0.30 per run at Opus 4.7 pricing
- QA Agent: ~5-10 tool calls (read diff + write to Linear) → ~$0.03-0.08 per review
- Builder Agent: ~10-20 tool calls (read files, edits, run validations) → ~$0.05-0.15 per task
- Git Agent: ~3-5 bash commands → ~$0.01-0.02 per invocation
- **Caching**: Enable prompt caching on the system prompt for all agents (static content — Anthropic beta cache headers). Reduces per-run cost by ~60-80% on repeated invocations.

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail) | Measurement Approach | Priority |
|-----------|-------------------|---------------------|----------|
| Capability boundary compliance | Pass: agent uses only allowed tools and produces no forbidden side effects (git writes by Builder, Linear writes by Builder/Git, code edits by QA/Git) | Code — inspect tool_use blocks in response; integration test asserts no forbidden API calls | Critical |
| Contract schema validity | Pass: output Pydantic model validates without exception; all required fields present with correct types | Code — `model.model_validate(output)` in test harness; ValidationError → fail | Critical |
| QA cycle cap enforcement | Pass: at qa_cycle_count==3, status is "approved" with tech_debt_annotation; no 4th fix cycle requested | Code — unit test with qa_cycle_count=3 input; assert output.qa_status == "approved" | Critical |
| Linear hierarchy correctness | Pass: EPIC → User Story (≥1) → Task (≥1 per story) → Subtask created in Linear; parent IDs correct; plan.md quote in description | LLM Judge + Human — inspect created Linear items; judge evaluates hierarchy completeness against plan.md | High |
| Double-claim prevention | Pass: re-running Backlog Agent on same plan.md does not create duplicate EPICs in Linear | Code — integration test: run twice, count EPICs; assert count unchanged on second run | High |
| Fix subtask count cap | Pass: QA creates 0–5 subtasks per report; never 6+ | Code — count created subtasks in Linear after QA run; assert len <= 5 | High |
| Branch and PR naming | Pass: branch is `feature/LIN-{id}-{slug}`, PR title is `[LIN-{id}] {description}`, PR targets EPIC branch | Code — `gh pr view --json headRefName,title,baseRefName`; regex match | High |
| REBASE_STACK completeness | Pass: all open task PRs targeting EPIC branch are rebased after a sibling merges; no sibling left with stale base | Integration test — create 2 sibling PRs, merge one, assert both rebase events triggered | Medium |
| Builder implementation scope | Pass: Builder modifies only files within the Linear work item scope; no out-of-scope file edits | LLM Judge — compare `git diff --name-only` against work item description; judge flags out-of-scope files | High |
| Instruction following | Pass: agent completes stated task from input contract without requesting clarification or hallucinating requirements not in the contract | LLM Judge — evaluate response against input contract; flag fabricated constraints | Medium |

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Capability boundary enforcement | Agent attempts to call a tool outside its scoped-tools list | Block — Claude Code skill runner refuses the tool call; agent must produce output contract without it |
| Contract schema validation | Agent output fails Pydantic `model_validate()` | Block — CLI raises ValidationError with field-level detail; operator must inspect and re-invoke |
| QA cycle cap | Input contract has qa_cycle_count >= 3 AND status would be changes_required | Block — Pydantic `model_validator` raises; forces "approved + tech_debt_annotation" output |
| Fix subtask count | QA output has > 5 findings | Block — Pydantic `Field(max_length=5)` on findings list; agent must consolidate findings |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| Hierarchy correctness rate | Review Linear workspace after every Backlog Agent run; flag if EPIC has no User Stories | Inspect plan.md chunking logic; adjust system prompt few-shot example |
| Capability boundary compliance rate | Automated: check tool_use blocks in Phoenix traces for forbidden tool names | Fix SKILL.md allowed-tools frontmatter; re-run integration tests |
| Contract validation failure rate | Count ValidationError exceptions in CLI logs per week | Update Pydantic model to match AGENT-CONTRACTS.md; add model_validator for new constraint |

## 7. Production Monitoring

**Tracing Tool:** Arize Phoenix (open-source, self-hosted locally for solo dev; no external infrastructure needed)

**Key Metrics to Track:**
1. Tool calls per agent run (alert if Backlog Agent exceeds 50 — likely infinite loop)
2. Pydantic validation failure rate per agent (should be 0%; any failure indicates schema drift)
3. QA cycle count distribution (alert if qa_cycle_count=3 is >20% of reviews — indicates systemic quality issues)
4. Linear items created per Backlog run (alert if 0 — indicates plan.md parsing failure)
5. REBASE_STACK coverage (fraction of sibling PRs successfully rebased per trigger event)

**Alert Thresholds:**
- Tool call count > 50 per Backlog Agent run → likely infinite loop, kill and inspect
- Pydantic validation failures > 0 in any 24h window → immediate investigation
- QA cycle_count = 3 (cap hit) > 3 times per week → escalate to operator (systemic quality signal)

**Smart Sampling Strategy:**
- Always inspect: any run where a Pydantic ValidationError was raised
- Sample 100%: first 10 production runs of each agent (establish baseline behavior)
- Sample 25%: ongoing Backlog Agent runs (hierarchy correctness spot-check)
- Sample on signal: QA Agent runs where qa_cycle_count = 3 hit (always review these)
- Skip: Git Agent runs with successful branch/PR creation and no rebase events (deterministic, low signal)
