# 11 - KNOWLEDGE STORAGE (HSBTech)

## Objective
Persist long-term engineering knowledge generated by the system so it can be reused for future planning, implementation, QA, and architectural decisions.

---

## Core Principle

> Linear stores operational state. Knowledge Storage stores reusable intelligence.

---

## Persistence Model

### 1. Operational Memory

Stored in Linear.

Use for:
- Current workflow state
- Issue comments
- QA findings linked to tasks
- PR references
- Fix subtasks
- Blockers
- Decisions tied to a specific EPIC, User Story, Task, or Subtask

Linear remains the source of truth for active execution.

---

### 2. Analytical / Long-Term Memory

Stored in a dedicated Knowledge Store.

Use for:
- Recurring QA issues
- Architectural decisions
- Reusable implementation patterns
- Codebase hotspots
- Historical risks
- Cross-EPIC insights
- Lessons learned
- Anti-patterns discovered over time

The Knowledge Store improves future decision-making.

---

## Storage Strategy

Recommended MVP options:

1. Markdown files versioned in the repository
2. SQLite/Postgres structured tables
3. Vector database for semantic retrieval
4. Hybrid model: structured DB + vector retrieval

## Data Categories

### Architecture Knowledge

Examples:
- preferred layering strategy
- service boundaries
- dependency rules
- architectural constraints

### QA Knowledge

Examples:
- recurring failure types
- risky modules
- repeated test gaps
- regression-prone areas

### Implementation Knowledge

Examples:
- common patterns
- reusable helpers
- common fixes
- coding conventions discovered from practice

### Backlog Knowledge

Examples:
- task decomposition rules

- common planning gaps

---

## Ingestion Rules

Knowledge should be persisted when:

- A QA finding repeats
- A PR exposes architectural drift
- A workaround becomes a reusable pattern
- A recurring implementation mistake is detected
- A backlog decomposition problem repeats
- A decision should influence future work

---

## Retrieval Rules

Knowledge should be retrieved before:

- Backlog Planning
- Implementation
- QA Review
- UAT Validation
- Architectural decision-making
- Large refactors

---

## Integration Points

### With Knowledge / Context Enrichment

This skill provides the storage layer used by Knowledge Enrichment.

### With QA Review

QA findings may be promoted to long-term knowledge when they represent recurring issues or systemic risks.

### With Backlog Planning

Historical decomposition lessons should improve future EPIC/User Story/Task generation.

### With Implementation

Known patterns and anti-patterns should guide implementation strategy.

---

## Quality Guidelines (Harness)

- Store only reusable knowledge
- Avoid storing noisy or one-off details
- Include evidence whenever possible
- Keep entries concise and actionable
- Prefer structured entries over free-form notes

---

## Anti-patterns

- Duplicating all Linear comments into Knowledge Store
- Persisting temporary state as long-term knowledge
- Storing vague insights without evidence
- Creating knowledge entries that cannot be reused
- Using Knowledge Store as the execution state

---

## Golden Rule

> If it affects the current workflow, store it in Linear. If it improves future workflows, store it in Knowledge Storage.
